## 1. 프로젝트 개요

OneTakeStudio는 실시간 라이브 스트리밍과 AI 쇼츠 생성을 지원하는 통합 영상 송출 솔루션이다. 2-Service MSA 구조(Core Service + Media Service)를 기반으로, LiveKit WebRTC를 통한 실시간 송출과 녹화, FastAPI 기반 AI 서비스를 통한 자동 쇼츠 생성 기능을 제공한다.

**아키텍처 구성**
- Core Service: 사용자 인증, 스튜디오 관리, 라이브러리, 알림
- Media Service: 스트리밍, 녹화, 채팅, 멀티 플랫폼 송출
- AI Service: 영상 분석 및 쇼츠 자동 생성 (FastAPI)
- Frontend: Next.js 기반 SPA

## 5. 데이터베이스 설정

### 5.1 MySQL 초기화

MySQL 8.0을 설치한 뒤 아래 스크립트를 실행하여 DB와 사용자를 생성한다.

```sql
CREATE DATABASE IF NOT EXISTS core_db
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE DATABASE IF NOT EXISTS media_db
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE USER IF NOT EXISTS 'user_core'@'%' IDENTIFIED BY 'core_password';
GRANT ALL PRIVILEGES ON core_db.* TO 'user_core'@'%';

CREATE USER IF NOT EXISTS 'user_media'@'%' IDENTIFIED BY 'media_password';
GRANT ALL PRIVILEGES ON media_db.* TO 'user_media'@'%';

FLUSH PRIVILEGES;
```

전체 테이블 스키마는 `exec/db_dump.sql` 파일을 참조한다.

### 5.2 JPA DDL 전략

두 서비스 모두 `spring.jpa.hibernate.ddl-auto: update`를 사용하므로, 애플리케이션을 처음 실행하면 엔티티 기반으로 테이블이 자동 생성된다. 별도의 마이그레이션 도구(Flyway, Liquibase)는 사용하지 않는다.

### 5.3 DB 접속 정보

**로컬 개발**

| 항목 | core_db | media_db |
|------|---------|----------|
| Host | localhost | localhost |
| Port | 3306 | 3306 |
| Database | core_db | media_db |
| Username | core_user | media_user |
| Password | core_password | media_password |

**운영 (Kubernetes)**

| 항목 | 값 |
|------|-----|
| Host | mysql.database.svc.cluster.local |
| Port | 3306 |
| 계정 | K8s Secret (db-secret)으로 관리 |

## 6. 인프라 구성

### 6.1 Redis

**설치 (로컬)**
```bash
# Docker로 실행
docker run -d --name redis -p 6379:6379 redis:7-alpine
```

**설정값** (운영 환경 ConfigMap)
```
appendonly yes
appendfsync everysec
maxmemory 200mb
maxmemory-policy allkeys-lru
```

**용도**
- JWT 블랙리스트 (로그아웃 처리)
- 이메일 인증 코드 임시 저장
- Redis Streams: 서비스 간 이벤트 통신
  - `media-events`: Media → Core (RECORDING_STOPPED)
  - `live-stream-events`: Media → Core (LIVE_STARTED/ENDED)
- LiveKit 내부 통신 (Server ↔ Egress 간 작업 큐)

### 6.2 LiveKit

**구성 요소 3개**
1. **LiveKit Server** — WebRTC SFU 서버 (포트 7880, 7881, UDP 50200-50250)
2. **LiveKit Egress** — 녹화 및 RTMP 송출 (CPU 집약적)
3. **LiveKit Ingress** — 외부 RTMP 수신 (포트 1935)

**로컬 실행**: `docker-compose.yml`의 livekit, livekit-egress, livekit-ingress 서비스를 띄운다.

**API 인증**: API Key (`devkey`) + API Secret (32자 이상 문자열). 운영 환경에서는 별도 생성 필요.

## 7. 백엔드 빌드 및 실행

### 7.1 사전 조건
- JDK 21 설치
- Maven 3.9 설치 (또는 IntelliJ 내장 Maven 사용)
- MySQL 8.0, Redis 7 실행 중

### 7.2 빌드

```bash
cd onetakestudio-backend

# 전체 빌드 (테스트 스킵)
mvn clean package -DskipTests
```

모듈별 빌드 순서: common → core-service, media-service (병렬 가능)

```bash
# common + core-service만 빌드
mvn clean package -pl common,core-service -am -DskipTests

# common + media-service만 빌드
mvn clean package -pl common,media-service -am -DskipTests
```

### 7.3 실행

```bash
# Core Service (포트 8080)
java -jar core-service/target/core-service-0.0.1-SNAPSHOT.jar

# Media Service (포트 8082)
java -jar media-service/target/media-service-0.0.1-SNAPSHOT.jar

# (선택) Eureka Server (포트 8761) - 로컬에서는 불필요
java -jar eureka-server/target/eureka-server-0.0.1-SNAPSHOT.jar

# (선택) API Gateway (포트 60000) - 로컬에서는 불필요
java -jar api-gateway/target/api-gateway-0.0.1-SNAPSHOT.jar
```

로컬 개발 시 Eureka와 Gateway 없이도 동작한다. 프론트엔드의 next.config.ts가 직접 Core/Media Service로 프록시 처리한다.

### 7.4 운영 프로파일 실행

```bash
java -jar core-service/target/core-service-0.0.1-SNAPSHOT.jar \
  --spring.profiles.active=prod

java -jar media-service/target/media-service-0.0.1-SNAPSHOT.jar \
  --spring.profiles.active=prod
```

운영 프로파일(`application-prod.yml`)은 모든 민감 정보를 환경 변수에서 읽어온다.

Dockerfile은 `eclipse-temurin:21-jre-alpine` 베이스 이미지를 사용하며, FFmpeg를 Alpine 패키지로 설치한다.

## 13. 배포 시 특이사항

### 13.1 JWT Secret 동기화

Core Service, Media Service, API Gateway 세 곳 모두 동일한 JWT Secret을 사용해야 한다. 하나라도 다르면 토큰 검증이 실패한다. 운영에서는 K8s Secret으로 관리하고, 로컬에서는 각 application.yml에 동일 값이 하드코딩되어 있다.

### 13.2 LiveKit API Secret 길이

LiveKit 1.9 이상에서는 API Secret이 최소 32자 이상이어야 한다. 짧으면 서버가 시작되지 않으므로 주의한다.

### 13.3 FFmpeg 경로 차이

- 로컬 (Windows): `AI/venv/Lib/site-packages/imageio_ffmpeg/binaries/ffmpeg-win-x86_64-v7.1.exe`
- Docker/K8s: `/usr/bin/ffmpeg` (alpine 패키지 설치)

Dockerfile에서 `apk add --no-cache ffmpeg`로 설치하므로 별도 설정 불필요. 로컬 Windows에서는 경로를 application.yml에 직접 지정해야 한다.

### 13.4 녹화 파일 공유 볼륨

Core Service와 AI Service가 동일한 녹화 파일에 접근해야 한다. K8s에서는 hostPath(`/data/recordings`)를 Core Service와 Egress가 공유한다. 다중 노드 환경에서는 NFS 또는 EFS로 교체해야 한다.

### 13.5 AI 서비스 배포

AI 서비스는 GPU가 필요하므로 K8s 내부가 아닌 별도 서버에서 실행 중이며, ngrok 터널을 통해 Core Service와 통신한다. 안정적 운영을 위해서는 GPU 노드 추가 또는 클라우드 GPU 인스턴스를 사용해야 한다.

### 13.6 Eureka 비활성화

운영 환경에서는 Eureka를 사용하지 않는다. Media Service의 `application-prod.yml`에 `eureka.client.enabled: false`가 설정되어 있고, 서비스 간 통신은 K8s 내부 DNS로 직접 접근한다.

### 13.7 CORS 설정

백엔드 SecurityConfig에서 `allowedOriginPatterns: *`로 모든 Origin을 허용하고 있다. Traefik Ingress에서도 별도의 CORS 미들웨어가 설정되어 있다. 프로덕션 배포 전 허용 도메인을 제한하는 것을 권장한다.
