# 백엔드 답변 반영 — 프론트엔드 검토 결과

> 작성일: 2026-01-29  
> 목적: 백엔드 답변 기준으로 구현 가능 여부·수정 포인트·추가 확인 사항 정리

---

## 1. 결론: **지금 정보만으로 구현 가능 여부**

| 구분                       | 판단         | 비고                                             |
| -------------------------- | ------------ | ------------------------------------------------ |
| **통합 비디오 목록/상세**  | ✅ 가능      | 응답 래퍼·id 타입·studioId만 반영하면 됨         |
| **업로드**                 | ✅ 가능      | Media-Service 경로·Form 스펙 확정됨              |
| **트림(Save) → 클립 생성** | ✅ 가능      | POST /api/library/clips로 대체                   |
| **클립 목록(사이드바)**    | ✅ 가능      | 상세 응답의 clips 또는 GET recordings/{id}/clips |
| **다운로드**               | ⚠️ 부분 가능 | 단건은 가능, 다건(체크박스)는 추후 확인 권장     |
| **AI 쇼츠 생성/폴링**      | ❌ 당장 불가 | AI 연동 전까지 MSW 유지                          |
| **댓글 타임라인**          | ❌ 당장 불가 | 추후 구현으로 스킵                               |

**요약:**

- 라이브러리 목록·상세·업로드·수동 클립(트림 저장)·클립 목록·단건 다운로드는 **현재 스펙으로 구현 가능**합니다.
- **studioId**를 어디서 넘길지만 정하면 됩니다.
- AI 쇼츠·댓글 타임라인은 백엔드/AI 쪽 일정에 맞춰 나중에 연동하면 됩니다.

---

## 2. 프론트엔드에서 반드시 수정·반영할 사항

### 2.1 응답 래퍼 `{ success, data }`

백엔드는 다음 형식을 사용합니다.

```json
{ "success": true, "data": { "videos": [...], "total": 8 } }
```

현재 프론트는 `apiClient.get(url, VideoListResponseSchema)`로 **전체 응답**을 검증하고 있어서, `data` 안의 `videos`/`total`을 기대하는 스키마와 맞지 않습니다.

**조치:**

- Library/Video API용으로 **ApiResponse 래퍼 스키마**를 두고, `data`만 꺼내서 기존 `VideoListResponseSchema` 등으로 검증하거나,
- 응답을 받은 뒤 `response.data.videos`, `response.data.total`처럼 **data만 사용**하도록 호출부에서 한 단계 언래핑합니다.  
  (다른 Core API(스튜디오 등)와 동일하게 `response.data`를 쓰는 방식 권장.)

---

### 2.2 비디오 ID 타입: `number` → `string` (UUID)

백엔드: **recordingId / clipId = UUID 문자열**  
현재 프론트: `VideoSchema`의 `id: z.number()` 및 `Video` 타입이 `id: number`로 되어 있음.

**조치:**

- `Video` 스키마의 `id`를 **`z.string()`** 로 변경.
- 상세·다운로드·클립 생성 등 모든 API에서 `videoId`/`recordingId`/`clipId`를 **문자열**로 처리.
- 라우트 `/library/[id]`는 이미 문자열이라 그대로 사용 가능.

---

### 2.3 비디오 목록 API에 `studioId` 필수

백엔드: `GET /api/library/videos?type=all&studioId=1` 형태로 **studioId 쿼리 필수**.

현재 프론트: `VideoLibrary`에서 `studioId`를 넘기지 않음.

**조치 (택 1):**

1. **워크스페이스/스튜디오 컨텍스트 사용**
   - 현재 선택된 스튜디오가 있다면 그 `studioId`를 쿼리에 포함.
2. **URL에서 스튜디오 지정**
   - 예: `/workspace/[userId]/library?studioId=xxx` 등으로 들어오면 해당 값을 사용.
3. **백엔드와 협의**
   - “라이브러리는 사용자 단위로만 쓴다”면 `userId`만으로 목록 조회 가능한 API가 있는지 확인 (있다면 studioId 불필요).

**구현 전에:** studioId를 **어디서 가져올지**(워크스페이스/URL/기본값 등) 한 번 정해야 합니다.

---

### 2.4 요청 헤더 `X-User-Id`

백엔드: `X-User-Id: {userId}` 사용.

프론트: `useAuthStore`의 `user.userId`가 문자열로 있음.

**조치:**

- Library/Video/Clip 관련 API 호출 시 **공통으로 `X-User-Id: user.userId`** 를 붙이기.
- `apiClient` 인터셉터에 추가하거나, Library 전용 래퍼에서 처리하면 됩니다.

---

### 2.5 트림 저장(Save) → 클립 생성 API로 대체

초안: “Save 버튼 = 트림 구간 저장”.  
백엔드: **PUT /api/library/videos/{videoId}/trim** 대신 **POST /api/library/clips**로 클립 생성.

**Request:**  
`{ recordingId, title, startTimeSec, endTimeSec }`

**조치:**

- 상세 페이지에서 “Save” 클릭 시,
  - 현재 선택한 구간: `startTimeSec`, `endTimeSec`,
  - 제목: 기존 영상 제목 + “(편집)” 등
  - 로 **POST /api/library/clips** 호출.
- `recordingId`는 현재 보고 있는 원본 비디오의 id(recordingId, 문자열)로 전달.

---

### 2.6 업로드 API 경로 및 Form

백엔드: **POST /api/v1/media/upload** (Media-Service)

- Form: `studioId`, `title`, `file`
- 헤더: `X-User-Id`

**조치:**

- 업로드 모달에서 **FormData**로 `studioId`, `title`, `file` 전송.
- 업로드 시 **studioId**가 필요하므로, 2.3에서 정한 “studioId 소스”와 동일하게 맞추기.

---

### 2.7 쇼츠(클립) 목록 조회

백엔드:

- 상세 응답의 `data.clips`
- 또는 **GET /api/library/recordings/{recordingId}/clips**

**조치:**

- 우측 사이드바의 “쇼츠 목록”은 상세 한 번 호출로 `clips`를 쓰거나, 필요 시 위 전용 API로 갱신.

---

### 2.8 상태 매핑

백엔드 → 프론트는 아래처럼 맞추면 됩니다.

| 백엔드           | 프론트     |
| ---------------- | ---------- |
| PROCESSING       | Processing |
| READY            | Saved      |
| FAILED           | Failed     |
| (업로드 직후 등) | Uploaded   |

필요하면 DTO 변환 레이어에서 한 번만 매핑하도록 두면 됩니다.

## 3. 추가로 확인하면 좋은 사항 (선택)

| 번호 | 내용                                                                                        | 현재 프론트 가정                                                                                                 |
| ---- | ------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| 1    | **다운로드 여러 개 선택** 시 (체크박스) API 형태                                            | 단건: GET /api/library/videos/{videoId}/download. 여러 개는 1) 여러 번 호출 2) ZIP 등 별도 API 있는지 확인 권장. |
| 2    | **GET /api/library/videos**에서 studioId 없이 “현재 사용자 전체” 목록을 주는 옵션 존재 여부 | 없으면 반드시 studioId 필요(위 2.3).                                                                             |
| 3    | **API Gateway**에 `/api/library/**` 라우트가 실제로 추가되었는지                            | Core-Service로 가도록 되어 있어야 프론트에서 그대로 호출 가능.                                                   |

## 4. 구현 순서 제안 (프론트)

1. **공통**
   - Video 스키마 `id` → string, ApiResponse 래퍼 처리, X-User-Id 헤더, studioId 소스 결정.
2. **비디오 목록**
   - GET /api/library/videos?type=all&studioId=xxx 연동, 래퍼·id 타입 반영.
3. **비디오 상세**
   - GET /api/library/videos/{videoId} 연동, 상세 정보 + clips로 사이드바 표시.
4. **업로드**
   - POST /api/v1/media/upload (FormData, studioId, title, file).
5. **트림(Save)**
   - POST /api/library/clips (recordingId, title, startTimeSec, endTimeSec).
6. **클립 목록**
   - 상세의 clips 또는 GET /api/library/recordings/{id}/clips.
7. **다운로드**
   - 단건: GET /api/library/videos/{videoId}/download (링크 또는 blob 처리).
8. **AI 쇼츠/댓글**
   - 백엔드/AI 준비되기 전까지 MSW 유지, 댓글 타임라인은 비활성 또는 플레이스홀더.

---

## 5. 요약

- **지금 정보만으로** 라이브러리 목록·상세·업로드·수동 클립(트림)·클립 목록·단건 다운로드까지는 **구현 가능**합니다.
- **꼭 할 일:**
  - 응답 `{ success, data }` 처리,
  - Video `id`를 string(UUID)로 통일,
  - `X-User-Id` 헤더,
  - **studioId** 조회 경로 확정.
- **선택 확인:** 다건 다운로드 방식, studioId 없이 사용자 전체 목록 지원 여부, Gateway `/api/library/**` 적용 여부.

이대로 반영하면 백엔드와의 충돌 없이 병합 가능한 상태로 구현할 수 있습니다.
