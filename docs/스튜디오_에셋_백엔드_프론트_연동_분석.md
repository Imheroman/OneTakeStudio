## 2. 백엔드 상세 분석

### 2.1 API 엔드포인트

| 메서드 | 경로                                       | 구현                |
| ------ | ------------------------------------------ | ------------------- |
| GET    | `/api/studios/{studioId}/assets`           | ✅                  |
| GET    | `/api/studios/{studioId}/assets?type=logo` | ✅ (type 쿼리 지원) |
| POST   | `/api/studios/{studioId}/assets`           | ⚠️ JSON만           |
| DELETE | `/api/studios/{studioId}/assets/{assetId}` | ✅                  |

### 2.2 CreateAssetRequest (POST 요청 스키마)

```java
// core-service/.../dto/CreateAssetRequest.java
private String type;   // "logo" | "overlay" | "video"
private String name;   // 필수, 최대 100자
private String fileUrl; // 선택 - URL 문자열만
```

- **Content-Type**: `application/json` 만 지원
- **MultipartFile**: 없음 → **실제 파일 업로드 불가**

### 2.3 StudioAsset 엔티티 (DB 저장 구조)

| 컬럼       | 타입         | 설명                  |
| ---------- | ------------ | --------------------- |
| id         | BIGINT (PK)  | 자동 증가             |
| asset_id   | VARCHAR(36)  | UUID, 고유 식별자     |
| studio_id  | BIGINT       | 스튜디오 FK           |
| type       | VARCHAR(20)  | LOGO, OVERLAY, VIDEO  |
| name       | VARCHAR(100) | 표시 이름             |
| file_url   | VARCHAR(500) | **URL 문자열만 저장** |
| file_size  | BIGINT       | (미사용)              |
| sort_order | INT          | 정렬 순서             |
| created_at | DATETIME     | 생성 시각             |

### 2.4 데이터 저장 위치

| 데이터 종류              | 저장 위치                      | 비고                     |
| ------------------------ | ------------------------------ | ------------------------ |
| **이미지 (PNG, JPG 등)** | ❌ 저장 안 됨                  | 파일 업로드 API 없음     |
| **비디오 (MP4 등)**      | ❌ 저장 안 됨                  | 파일 업로드 API 없음     |
| **텍스트 (이름)**        | ✅ DB `studio_assets.name`     |                          |
| **fileUrl**              | ✅ DB `studio_assets.file_url` | **외부 URL만** 저장 가능 |

**결론**: 백엔드는 **파일을 저장하지 않습니다**. `fileUrl`에 **이미 업로드된 파일의 URL**을 넣어야 합니다.  
→ 이미지/비디오를 직접 업로드하려면 **별도 파일 업로드 서비스**가 필요합니다.

## 3. 프론트엔드 현황

### 3.1 API 클라이언트 (`studio-assets.ts`)

| 함수                                                    | 구현    | 백엔드 호환               |
| ------------------------------------------------------- | ------- | ------------------------- |
| `getStudioAssets(studioId)`                             | ✅      | ✅                        |
| `createStudioAsset(studioId, { type, name, fileUrl? })` | ✅ JSON | ✅                        |
| `deleteStudioAsset(studioId, assetId)`                  | ✅      | ⚠️ assetId 타입 확인 필요 |

### 3.2 에셋 패널 (`StudioAssetPanel.tsx`)

- 목록 조회, 삭제: 구현됨
- **추가 버튼**: UI만 있고 **클릭 핸들러 없음** (기능 미구현)
  - "+ 로고 추가", "+ 인트로", "+ 아웃트로" → 클릭해도 동작 없음

### 3.3 프리뷰 연동 (`PreviewArea.tsx`)

- `AssetOverlayNode`: `asset.fileUrl`이 있으면 `<img>` 또는 비디오로 표시
- 드래그·리사이즈 지원

## 4. 프론트-백엔드 연동 가능성

### 4.1 현재 바로 연동 가능한 것

| 기능                                    | 상태         |
| --------------------------------------- | ------------ |
| 에셋 목록 조회                          | ✅ 연동 가능 |
| 에셋 삭제                               | ✅ 연동 가능 |
| **URL만 등록** (외부 이미지/비디오 URL) | ✅ 연동 가능 |

### 4.2 추가 구현이 필요한 것

| 기능                        | 필요 작업                                    |
| --------------------------- | -------------------------------------------- |
| **로고/이미지 파일 업로드** | 백엔드: multipart API + 파일 저장소(S3/로컬) |
| **비디오 파일 업로드**      | 동일                                         |
| 에셋 추가 UI                | 프론트: 모달/폼 + `createStudioAsset` 호출   |

### 4.3 ID 타입 불일치

- **백엔드 DELETE**: `@PathVariable Long assetId` → 숫자 `id` 사용
- **프론트**: `AssetItem.id`를 `String`으로 사용, `mapAssetResponseToItem`에서 `String(a.id)` 변환
- **AssetResponse**: `id`가 `Long` (숫자)
- **결론**: 프론트에서 `deleteStudioAsset(studioId, assetId)` 호출 시 **숫자 id**를 넘기면 됨. 현재 `String(a.id)`로 변환해서 넘기므로 `"1"` 같은 문자열이 전달됨. Spring이 `Long`으로 파싱 가능하므로 **문제 없음**.

## 5. 에셋 로직 흐름

```
[프론트]                    [백엔드]                    [저장소]
   │                          │                          │
   │  GET /assets             │                          │
   │ ───────────────────────> │  DB 조회                 │
   │                          │  studio_assets 테이블    │
   │  <───────────────────────│  (id, type, name,       │
   │  [{ id, type, name,      │   file_url, ...)         │
   │    fileUrl }]            │                          │
   │                          │                          │
   │  POST /assets            │                          │
   │  { type, name, fileUrl } │  DB INSERT               │
   │ ───────────────────────> │  file_url = request.    │
   │                          │         getFileUrl()     │
   │                          │  (파일 저장 없음!)      │
   │  <───────────────────────│                          │
   │  생성된 에셋             │                          │
   │                          │                          │
   │  DELETE /assets/{id}     │  DB DELETE              │
   │ ───────────────────────> │  (파일 삭제 없음)       │
```

**핵심**: `fileUrl`은 **클라이언트가 제공한 URL**을 그대로 DB에 저장합니다.  
파일을 서버에 업로드하는 로직은 **없습니다**.

## 6. 권장 사항

### 6.1 단기 (현재 구조로 가능)

1. **URL만 등록하는 에셋 추가 UI** 구현

   - 모달: type, name, fileUrl 입력
   - `createStudioAsset(studioId, { type, name, fileUrl })` 호출
   - 예: 사용자가 이미지 호스팅 URL(Imgur, S3 presigned URL 등)을 붙여넣기

2. **기본 에셋** (프론트 하드코딩)
   - 문서에 명시된 대로 프론트에서 기본 로고/오버레이를 섞어서 표시

### 6.2 중장기 (파일 업로드 지원)

1. **백엔드**

   - `MultipartFile` 수신 API 추가
   - 파일 저장: media-service의 S3/로컬 스토리지 활용 또는 core-service에 스토리지 모듈 추가
   - 저장 후 `fileUrl` 생성하여 `StudioAsset`에 저장

2. **프론트**
   - `FormData`로 파일 업로드
   - `apiClient.post`에 `Content-Type` 제거 (브라우저가 multipart boundary 자동 설정)

## 7. 결론

| 질문                                | 답변                                                                                 |
| ----------------------------------- | ------------------------------------------------------------------------------------ |
| **로고(이미지) 파일 업로드 가능?**  | ❌ 불가. URL만 등록 가능                                                             |
| **텍스트만 가능?**                  | ✅ name(표시 이름) 저장 가능. 단, fileUrl 없으면 프리뷰에 표시할 이미지/비디오 없음  |
| **비디오 업로드 가능?**             | ❌ 불가. URL만 등록 가능                                                             |
| **프론트 바로 연동 가능?**          | ✅ 목록/삭제/URL 등록은 가능. **파일 업로드**는 백엔드 확장 필요                     |
| **이미지/텍스트/비디오 저장 위치?** | **DB `studio_assets.file_url`에 URL만 저장**. 실제 파일은 **어디에도 저장되지 않음** |
