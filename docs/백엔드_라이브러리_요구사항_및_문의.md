# 라이브러리 페이지 - 백엔드 요구사항 및 문의

> 작성일: 2026-01-29  
> 작성: 프론트엔드  
> 목적: 라이브러리 페이지 구현을 위한 백엔드 API/DB 요구사항 정리 및 협의

---

## 📌 1. 배경

라이브러리 페이지(피그마 초안 기준) 구현을 위해 프론트엔드에서 필요한 백엔드 API와 DB 구조를 정리했습니다.  
**현재 백엔드 작업 현황 및 계획**을 알려주시면, 프론트엔드에서 그에 맞춰 API 연동을 진행하겠습니다.

## 📌 2. 백엔드 수정/구현 포인트 요약

### 2.1 DB 관련

| 항목                     | 내용                                       | 현재 상태                      |
| ------------------------ | ------------------------------------------ | ------------------------------ |
| **clips 테이블**         | 쇼츠(하이라이트 클립) 저장                 | ERD 문서에만 존재, 코드 미구현 |
| **로컬 업로드 영상**     | 사용자가 직접 업로드한 영상 저장           | 별도 테이블 없음               |
| **라이브러리 통합 조회** | 원본(녹화/업로드) + 쇼츠를 하나의 목록으로 | -                              |

**문의 1)**

- `clips` 테이블 구현 예정이신가요? 예정이라면 예상 스키마(컬럼)를 알려주실 수 있을까요?
- 로컬 업로드 영상은 `recordings` 확장 vs `uploaded_videos` 신규 테이블 중 어떤 방향으로 가실 계획인가요?

| `/api/v1/shorts/**` | Media 또는 별도 | 라우트 없음 |

**문의 2)**

- Library API는 Core Service vs Media Service 중 어디에서 담당하실 예정인가요?

### 2.3 Library API (비디오 목록/상세/업로드/다운로드/트림)

| API         | 메서드                                 | 용도                                    | 프론트 요청 형식                      |
| ----------- | -------------------------------------- | --------------------------------------- | ------------------------------------- |
| 비디오 목록 | `GET /api/library/videos`              | 갤러리 목록 (필터: all/original/shorts) | `?type=all` \| `original` \| `shorts` |
| 비디오 상세 | `GET /api/library/videos/:id`          | 상세 페이지 데이터                      | `:id` = 비디오 ID                     |
| 영상 업로드 | `POST /api/library/videos/upload`      | 로컬 파일 업로드                        | `multipart/form-data` (file)          |
| 트림 저장   | `PUT /api/library/videos/:id/trim`     | 영상 길이 조절 구간 저장                | `{ startTimeSec, endTimeSec }`        |
| 다운로드    | `GET /api/library/videos/:id/download` | 영상 파일 다운로드                      | `?ids=1,2,3` (여러 개 선택 시)        |

**문의 3)**

- 위 API 경로(`/api/library/...`)로 진행해도 될까요? 다른 경로 규칙이 있다면 알려주세요.
- 비디오 ID는 `recording_id`(UUID)를 사용할 예정인가요, 아니면 별도 `video_id`를 부여하실 예정인가요?

---

### 2.4 Shorts API (쇼츠 생성/상태/저장)

| API            | 메서드                         | 용도              | 프론트 요청 형식                               |
| -------------- | ------------------------------ | ----------------- | ---------------------------------------------- |
| 쇼츠 생성 요청 | `POST /api/v1/shorts/generate` | AI 쇼츠 생성 시작 | `{ videoId, bgColor, useSubtitles, language }` |
| 쇼츠 생성 상태 | `GET /api/v1/shorts/status`    | 폴링용 (1초 간격) | -                                              |
| 쇼츠 저장      | `POST /api/v1/shorts/save`     | 생성된 쇼츠 저장  | `{ clipIds: [1, 2, 3] }`                       |

**문의 4)**

- 쇼츠 생성은 AI 담당자와 연동이 필요할 것 같습니다. API 스펙(요청/응답)은 언제 확정될 예정인가요?
- 쇼츠 상태 폴링 시 `videoId` 기준으로 조회할 예정인가요? (현재 프론트는 한 번에 하나의 비디오에 대한 쇼츠만 생성한다고 가정)

## 📌 3. 프론트엔드가 기대하는 응답 형식 (참고)

백엔드에서 다른 형식을 사용하시면, 프론트에서 맞춰서 수정하겠습니다.  
아래는 현재 MSW 모킹 기준으로 사용 중인 형식입니다.

### 3.1 비디오 목록 `GET /api/library/videos`

```json
{
  "videos": [
    {
      "id": 1,
      "title": "Weekly Podcast Episode #45",
      "date": "Jan 15, 2026",
      "duration": "42:18",
      "type": "original",
      "status": "Uploaded",
      "thumbnailUrl": "https://..."
    },
    {
      "id": 4,
      "title": "Live Stream Highlight Reel",
      "date": "Jan 10, 2026",
      "duration": "8:52",
      "type": "shorts",
      "status": "Saved",
      "thumbnailUrl": "https://..."
    }
  ],
  "total": 8
}
```

- `type`: `"original"` | `"shorts"`
- `status`: `"Uploaded"` | `"Saved"` | `"Processing"` | `"Failed"`
- `duration`: `"42:18"` 형식 (MM:SS 또는 HH:MM:SS)

### 3.2 비디오 상세 `GET /api/library/videos/:id`

```json
{
  "id": 1,
  "title": "Weekly Podcast Episode #45",
  "date": "Jan 15, 2026, 03:16 PM",
  "duration": "42:18",
  "description": "Weekly podcast discussing...",
  "videoUrl": "https://s3.../video.mp4",
  "thumbnailUrl": "https://...",
  "clips": [
    {
      "id": 1,
      "title": "Shorts 1",
      "duration": "0:45",
      "url": "https://s3.../clip1.mp4",
      "thumbnailUrl": "https://..."
    }
  ]
}
```

- `clips`: 해당 원본에서 생성된 쇼츠 목록 (우측 사이드바용)

### 3.3 쇼츠 생성 요청 `POST /api/v1/shorts/generate`

**Request:**

```json
{
  "videoId": "1",
  "bgColor": "black",
  "useSubtitles": true,
  "language": "ko"
}
```

- `language`: `"ko"` | `"en"` | `"ja"` | `"zh"`

### 3.4 쇼츠 상태 `GET /api/v1/shorts/status`

**Response (폴링용):**

```json
{
  "status": "processing",
  "completedCount": 2,
  "videoId": "1"
}
```

- `status`: `"idle"` | `"processing"` | `"completed"`
- `completedCount`: 완료된 쇼츠 개수 (1~3)
- 완료 시 프론트에서 생성된 쇼츠 목록(URL 등)을 받을 수 있는 별도 API가 필요할 수 있습니다.

**문의 6)**

- 쇼츠 생성 완료 시, 각 쇼츠의 `url`, `thumbnailUrl` 등을 받는 API가 필요합니다.  
  `GET /api/v1/shorts/status` 응답에 포함할 예정인가요, 아니면 `GET /api/library/videos/:id/clips` 같은 별도 API를 사용할 예정인가요?

## 📌 4. 백엔드 답변 요청 사항 (체크리스트)

아래 항목에 대해 답변 주시면, 프론트엔드에서 그에 맞춰 구현하겠습니다.

| 번호 | 문의 내용                                                | 답변 |
| ---- | -------------------------------------------------------- | ---- |
| 1    | clips 테이블 구현 예정? 스키마? 로컬 업로드 테이블 방향? |      |
| 2    | Library/Shorts API 담당 서비스 (Core vs Media)?          |      |
| 3    | API 경로 규칙, 비디오 ID 체계?                           |      |
| 4    | 쇼츠 생성 API 스펙 확정 시기? videoId 기준 폴링?         |      |
| 5    | 댓글 타임라인 API 우선순위?                              |      |
| 6    | 쇼츠 생성 완료 시 URL 등 조회 API?                       |      |

### 추가로 알려주시면 좋은 것

- **현재 백엔드에서 작업 중인 라이브러리/쇼츠 관련 사항**
- **구현 예정인 API 목록 및 예상 일정**
- **응답 형식이 위와 다를 경우** → 실제 사용할 스펙
- **인증**: `Authorization: Bearer {token}` + `X-User-Id` 등 헤더 규칙

---

## 📌 5. 프론트엔드 진행 방식

1. 백엔드 답변 수신 후, 위 스펙에 맞춰 프론트 API 연동 코드 수정
2. API 미구현 구간은 MSW 모킹 유지 (개발/테스트용)
3. 백엔드 API 준비 시점에 MSW 제거 후 실제 API 연동으로 전환

## 📌 (추가) 백엔드 답변 반영 요약 (2026-01-29)

| 문의                 | 답변                                                      |
| -------------------- | --------------------------------------------------------- |
| 1. clips·로컬 업로드 | clips 테이블 존재, 로컬 업로드는 recordings.filePath 사용 |
| 2. 담당 서비스       | Library API → Core-Service, 파일 처리 → Media-Service     |
| 3. API·ID            | /api/library/videos 통합, ID는 UUID 문자열                |
| 4. 쇼츠 생성         | AI 연동 필요, 수동 Clip 생성 구현 예정                    |
| 5. 댓글 타임라인     | 우선순위 낮음, 추후                                       |
| 6. 쇼츠 URL 조회     | GET /api/library/recordings/{id}/clips                    |

- 응답 형식: `{ success, data }` 래퍼 사용
- 비디오 목록: `GET /api/library/videos?type=all&studioId=1`, 헤더 `X-User-Id`
- 상세: `GET /api/library/videos/{videoId}`
- 업로드: `POST /api/v1/media/upload` (Form: studioId, title, file)
- 트림(클립 생성): `POST /api/library/clips` (recordingId, title, startTimeSec, endTimeSec)
- 클립 목록: `GET /api/library/recordings/{recordingId}/clips`
