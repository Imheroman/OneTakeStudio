# 숏츠 저장 + 트림 기반 생성 + 스토리지 연동 구현

> 작업일: 2026-02-05
> 브랜치: dev

---

## 개요

숏츠 기능에 세 가지 주요 변경을 적용했다.

1. **트림 구간 기반 숏츠 생성** — 영상 길이 조절에서 선택한 구간을 잘라서 AI에 전달
2. **스토리지 연동** — 숏츠 파일 크기를 스토리지 사용량에 포함, 제한 10GB로 변경
3. **미저장 숏츠 자동 정리** — 24시간 지난 미저장 숏츠 파일 삭제 + EXPIRED 처리

---

## 파트 A: 트림 구간 기반 숏츠 생성

### 변경 흐름

```
비디오 상세 페이지에서 트림 구간 선택
  -> "Generate Shorts" 클릭
  -> query param으로 start/end 전달 (?start=150&end=525)
  -> 숏츠 설정 페이지에서 "선택된 구간: 2:30 ~ 8:45 (6분 15초)" 표시
  -> API 요청 시 trimStartSec, trimEndSec 포함
  -> 백엔드에서 FFmpeg으로 트림 -> 트림된 영상 기준 segmentation
```

### 백엔드 변경

#### `ShortsGenerateRequest.java`
- `trimStartSec` (Double, optional) 필드 추가
- `trimEndSec` (Double, optional) 필드 추가

#### `AiShortsService.java` — `requestGeneration()`
- 트림 파라미터가 있으면 먼저 FFmpeg으로 트림 구간을 잘라 임시 파일 생성
- 잘린 영상의 duration 기준으로 기존 segmentation 로직 실행
  - 10분 미만: 전체를 AI에 전송
  - 10분 이상: 3등분
- 트림 모드에서는 마커 기반 모드 비활성화 (트림된 영상에서 원본 마커 타임스탬프는 의미 없음)
- `calculateEvenSegmentsFromTrimmed()` 메서드 추가

### 프론트엔드 변경

#### `video-info-section.tsx`
- `onGenerateShorts` 콜백 prop 추가 (optional)
- 콜백이 있으면 사용, 없으면 기존 router.push 폴백

#### `video-detail-viewer.tsx`
- `handleGenerateShorts` 함수 추가
- 현재 trimStart/trimEnd가 전체 영상과 다르면 query param으로 전달
- `VideoInfoSection`에 `onGenerateShorts` prop 연결

#### `shorts/page.tsx`
- `useSearchParams()`로 `start`, `end` query param 파싱
- `ShortsConfigurator`에 `trimStartSec`, `trimEndSec` prop 전달

#### `shorts-configurator.tsx`
- props에 `trimStartSec?`, `trimEndSec?` 추가
- 트림 구간이 있으면 보라색 배너로 "선택된 구간: 2:30 ~ 8:45 (6분 15초)" 표시
- API 요청 body에 `trimStartSec`, `trimEndSec` 포함

## 파트 B: 스토리지 연동

### 변경 흐름

```
숏츠 생성 완료 (webhook)
  -> outputPath에서 File.length() 호출
  -> ShortsResult.fileSize에 저장
  -> getStorage() 호출 시 saved=true인 숏츠 파일 크기 합산
  -> 프론트엔드에서 shortsUsage 표시
```

### 백엔드 변경

#### `ShortsResult.java`
- `fileSize` (Long) 컬럼 추가

#### `AiShortsService.java` — `handleWebhook()`
- 숏츠 생성 완료 시 `outputPath` 파일의 크기를 `fileSize`에 저장

#### `ShortsResultRepository.java`
- `getTotalSavedFileSizeByUserId(Long userId)` — saved=true인 숏츠 파일 크기 합계 쿼리 추가

#### `LibraryService.java`
- `ShortsResultRepository`, `UserRepository` 주입
- `getStorage()`: 기존 `recordingSize + clipSize`에 `shortsSize` 추가
- 스토리지 제한: `32212254720` (30GB) -> `10737418240` (10GB)

#### `StorageResponse.java`
- `shortsUsage` (Double, GB 단위) 필드 추가
- `of(usedBytes, limitBytes, videoBytes, assetBytes, shortsBytes)` 5-파라미터 팩토리 메서드 추가

### 프론트엔드 변경

#### `dto/library.ts`
- `StorageResponseSchema`에 `videoUsage`, `assetUsage`, `shortsUsage` 필드 추가
- `StorageDataFromApiDto`에 `shortsUsage` 필드 추가

#### `library.ts` — `getStorage()`
- 기존: `videoUsage: 0, assetUsage: 0` 하드코딩
- 변경: 서버 응답의 `videoUsage`, `assetUsage`, `shortsUsage` 값 사용

## 파트 C: 미저장 숏츠 자동 정리

### 변경 흐름

```
매일 새벽 3시 @Scheduled 실행
  -> 24시간 이전 saved=false & status=COMPLETED 숏츠 조회
  -> 파일 삭제 (outputPath 기반)
  -> DB 상태를 EXPIRED로 변경, outputPath/fileSize 초기화
```

### 백엔드 변경

#### `ShortsResultStatus.java`
- `EXPIRED` enum 값 추가

#### `ShortsResultRepository.java`
- `findExpiredUnsaved(LocalDateTime cutoff)` — 정리 대상 숏츠 조회 쿼리 추가

#### `AiShortsService.java`
- `cleanupExpiredShorts()` 메서드 추가
  - `@Scheduled(cron = "0 0 3 * * *")` — 매일 새벽 3시 실행
  - 파일 삭제 후 상태를 `EXPIRED`로 변경

## DB 마이그레이션 필요 사항

`shorts_results` 테이블에 컬럼 추가 필요:

```sql
ALTER TABLE shorts_results ADD COLUMN file_size BIGINT;
```

> `status` 컬럼은 이미 VARCHAR(20)이므로 `EXPIRED` 값은 별도 마이그레이션 없이 저장 가능.
