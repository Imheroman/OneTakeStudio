# 로그아웃 방식 비교 및 설계

**작성일**: 2026-01-28
**브랜치**: be-dev

## 1. JWT 로그아웃 방식 비교

JWT는 Stateless 토큰이므로 서버에서 직접 "무효화"할 수 없다. 이를 해결하기 위한 주요 방식 4가지를 비교한다.

### 방식 1: 클라이언트 측 토큰 삭제만

| 항목 | 내용 |
|------|------|
| 구현 방법 | 프론트엔드에서 localStorage/cookie의 토큰 삭제 |
| 장점 | 구현이 매우 간단, 서버 변경 없음 |
| 단점 | **토큰 탈취 시 만료까지 악용 가능**, 서버에서 강제 로그아웃 불가 |
| 보안 수준 | 낮음 |

### 방식 2: Redis 토큰 블랙리스트 (채택)

| 항목 | 내용 |
|------|------|
| 구현 방법 | 로그아웃 시 토큰의 JTI를 Redis에 저장, 요청마다 블랙리스트 확인 |
| 장점 | 즉시 무효화 가능, Redis TTL로 자동 정리, 기존 인프라(Redis) 활용 |
| 단점 | 매 요청마다 Redis 조회 1회 추가 |
| 보안 수준 | 높음 |

### 방식 3: DB 기반 토큰 버전 관리

| 항목 | 내용 |
|------|------|
| 구현 방법 | User 테이블에 `token_version` 컬럼 추가, 로그아웃 시 버전 증가, JWT에 버전 포함 |
| 장점 | 전체 세션 강제 로그아웃 가능 (버전 1회 증가로 모든 토큰 무효) |
| 단점 | **매 요청마다 DB 조회 필요** (Redis보다 느림), 개별 토큰 단위 무효화 불가 |
| 보안 수준 | 중간 |

### 방식 4: Refresh Token만 서버 관리

| 항목 | 내용 |
|------|------|
| 구현 방법 | Refresh Token을 DB/Redis에 저장, 로그아웃 시 삭제. Access Token은 방치 |
| 장점 | Access Token 검증에 추가 비용 없음 |
| 단점 | **Access Token 만료(1시간)까지 여전히 유효**, 즉시 무효화 불가 |
| 보안 수준 | 중간 |

## 2. Redis 블랙리스트를 채택한 이유

### 2-1. 보안 요구사항 충족

OneTakeStudio는 라이브 스트리밍 플랫폼으로, 방송 제어/녹화/송출 등 민감한 작업을 수행한다. 토큰 탈취 시 **즉시 차단**이 필요하다.

- 방식 1 (클라이언트 삭제): 탈취된 토큰이 1시간 동안 유효 → **부적합**
- 방식 4 (Refresh만 관리): Access Token이 1시간 유효 → **부적합**
- 방식 3 (DB 버전): 즉시 차단 가능하나, 개별 토큰 무효화 불가, DB 조회 부담
- **방식 2 (Redis 블랙리스트): 즉시 차단 + 개별 토큰 단위 무효화 + 빠른 조회**

### 2-2. 기존 인프라 활용

프로젝트에 이미 Redis(포트 6379)가 Docker Compose로 운영 중이다. 추가 인프라 구축 비용이 없다.

### 2-3. 성능

| 방식 | 요청당 추가 비용 |
|------|-----------------|
| Redis 블랙리스트 | Redis GET 1회 (~0.1ms) |
| DB 버전 관리 | MySQL SELECT 1회 (~1-5ms) |

Redis는 인메모리 저장소로, MySQL 대비 10~50배 빠른 응답 속도를 제공한다.

### 2-4. 자동 정리 (TTL)

블랙리스트에 등록된 JTI는 해당 토큰의 남은 만료 시간만큼만 Redis에 유지된다. 토큰이 자연 만료되면 Redis에서도 자동 삭제되므로, 별도 정리 배치가 필요 없다.

## 3. 구현 설계

### 3-1. JWT에 JTI (JWT ID) 클레임 추가

```
기존: { sub, email, nickname, type, iid, iat, exp }
변경: { sub, email, nickname, type, iid, jti, iat, exp }
```

- `jti`: UUID v4 형식의 고유 토큰 식별자
- Access Token과 Refresh Token 모두에 JTI 추가

### 3-2. 로그아웃 Flow

```
1. 클라이언트 → POST /api/auth/logout (Authorization: Bearer <accessToken>)
2. 서버: Access Token에서 JTI, 만료시간 추출
3. 서버: Redis에 "blacklist:{jti}" 키로 저장, TTL = 토큰 남은 만료시간
4. 서버: 200 OK 응답
5. 클라이언트: 로컬 토큰 삭제
```

### 3-3. 요청 검증 Flow

```
1. 요청 수신 → JWT 파싱 → JTI 추출
2. Redis에서 "blacklist:{jti}" 존재 여부 확인
3. 존재하면 → 인증 거부 (로그아웃된 토큰)
4. 존재하지 않으면 → 정상 처리
```

### 3-4. 수정 대상 파일

| 파일 | 변경 내용 |
|------|----------|
| `common/.../jwt/JwtUtil.java` | JTI 클레임 추가, `getJti()` 메서드 추가 |
| `core-service/pom.xml` | `spring-boot-starter-data-redis` 의존성 추가 |
| `core-service/application.yml` | Redis 연결 설정 추가 |
| `core-service/.../config/RedisConfig.java` | 신규: Redis StringRedisTemplate 설정 |
| `core-service/.../security/TokenBlacklistService.java` | 신규: 블랙리스트 등록/조회 서비스 |
| `core-service/.../security/JwtAuthenticationFilter.java` | 블랙리스트 확인 로직 추가 |
| `core-service/.../auth/service/AuthService.java` | `logout()` 메서드 추가 |
| `core-service/.../auth/controller/AuthController.java` | `POST /api/auth/logout` 엔드포인트 추가 |
