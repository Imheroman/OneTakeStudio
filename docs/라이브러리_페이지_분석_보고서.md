## 📌 2. 피그마 초안 vs 현재 구현 상태

### 2.1 첫 번째 사진: Video Library 갤러리 페이지

| 초안 요구사항 | 구현 상태 | 비고 |
|-------------|----------|------|
| 갤러리 형식으로 영상 표시 | ✅ 구현됨 | `VideoLibrary` + `VideoCard` |
| 원본/쇼츠 필터링 | ✅ 구현됨 | `VideoFilter` (모두표시, 원본영상, 쇼츠영상) |
| **Upload File 버튼** | ❌ **미구현** | 상단에 버튼 없음 |
| 원본 클릭 → 상세페이지 | ✅ 구현됨 | `router.push(/library/${video.id})` |
| **쇼츠 클릭 → 상세 + 쇼츠재생 팝업** | ❌ **미구현** | 현재 원본/쇼츠 동일하게 상세만 이동 |

**필요 작업:**
1. `VideoLibrary` 또는 `WorkspaceTopNav`에 "Upload File" 버튼 추가
2. `VideoCard` 클릭 시 `video.type === "shorts"`이면 상세페이지 + 쇼츠 재생 모달 동시 표시

## 📌 3. 백엔드 DB 및 API 분석

### 3.1 현재 DB 구조

#### Core Service (MySQL)
- `users`, `studios`, `studio_members`, `scenes`, `member_invites`, `connected_destinations` 등
- **라이브러리/비디오 관련 테이블 없음**

#### Media Service (PostgreSQL)
- `recording_sessions`: 스튜디오 녹화 세션 (LiveKit 연동)
  - `studio_id`, `user_id`, `s3_url`, `duration_seconds` 등
  - **라이브 스트리밍 녹화용** → 로컬 업로드 영상과는 별개
- **clips(쇼츠) 테이블/엔티티 없음** (ERD 문서에는 있으나 실제 코드에 미구현)

### 3.2 ERD 문서(database-erd.md) 기준

| 테이블 | 서비스 | 용도 | 실제 구현 |
|--------|--------|------|----------|
| recordings | Media | 녹화본 | `recording_sessions`로 구현 (스튜디오 녹화) |
| clips | Media | 클립/쇼츠 | ❌ 미구현 |
| markers | Media | 마커 | ❌ 미구현 |

### 3.3 데이터 저장 방안 제안

#### (1) 원본 영상
- **방송 녹화본**: `recording_sessions` (이미 존재) → studio_id 기준 조회
- **로컬 업로드**: 새 테이블 필요
  - 예: `uploaded_videos` (user_id, title, s3_key, s3_url, duration_sec, created_at)
  - 또는 `recordings` 테이블 확장 (source_type: 'recording' | 'upload')

#### (2) 쇼츠(Clips)
- ERD의 `clips` 테이블 구현
  - `recording_id` (원본 영상 FK)
  - `start_time_sec`, `end_time_sec`, `duration_sec`
  - `url`, `thumbnail_url`, `srt_url` (자막)
  - `status` (PROCESSING, READY, FAILED)
  - `ai_score` (하이라이트 점수)

#### (3) 라이브러리 통합 뷰
- 원본 + 쇼츠를 하나의 "비디오 목록"으로 보여주려면:
  - **Option A**: Core Service에 `library_videos` 뷰/테이블 (recordings + clips 조인)
  - **Option B**: Media Service에서 `GET /api/v1/library/videos` 제공 (recordings + clips 통합)
  - **Option C**: Core Service에 Library API 추가, Media Service Feign 호출

### 3.4 필요한 백엔드 API 목록

| API | 메서드 | 용도 | 구현 상태 |
|-----|--------|------|----------|
| `/api/library/videos` | GET | 비디오 목록 (원본+쇼츠, 필터) | ❌ MSW만 |
| `/api/library/videos/:id` | GET | 비디오 상세 | ❌ |
| `/api/library/videos/upload` | POST | 로컬 영상 업로드 | ❌ |
| `/api/library/videos/:id/trim` | PUT | 트림 구간 저장 | ❌ |
| `/api/library/videos/:id/download` | GET | 다운로드 | ❌ |
| `/api/v1/shorts/generate` | POST | 쇼츠 생성 요청 | ❌ MSW만 |
| `/api/v1/shorts/status` | GET | 쇼츠 생성 상태 (폴링) | ❌ MSW만 |
| `/api/v1/shorts/:id/save` | POST | 쇼츠 저장 | ❌ |
| 댓글 타임라인 API | GET | 시간별 댓글 수량 | ❌ |

### 3.5 API Gateway 라우트 추가 필요

```yaml
# api-gateway/application.yml 에 추가 필요
- id: library-service
  uri: lb://???  # core 또는 media 또는 신규 library-service
  predicates:
    - Path=/api/library/**
  filters:
    - StripPrefix=0

- id: shorts-service
  predicates:
    - Path=/api/v1/shorts/**
```

## 📌 4. 구현 우선순위 제안

### Phase 1: 백엔드 기반 구축
1. **DB**: `clips` 테이블 생성 (media-service)
2. **DB**: 로컬 업로드용 테이블 또는 recordings 확장
3. **API**: `GET /api/library/videos` (recordings + clips 통합)
4. **API**: `GET /api/library/videos/:id` (상세)
5. **API Gateway**: `/api/library/**` 라우트 추가

### Phase 2: 업로드 & 상세 페이지
1. **Upload File 버튼** + **업로드 모달** UI
2. **업로드 API** (`POST /api/library/videos/upload`)
3. **VideoPlayer** 실제 재생 + 컨트롤
4. **VideoInfoSection** API 연동 (제목, 날짜, 설명)
5. **VideoSidebar** API 연동 (원본 + 쇼츠 목록)

### Phase 3: 다운로드 & 편집
1. **다운로드 모달** (체크박스 선택 + 경로)
2. **다운로드 API**
3. **영상 길이 조절** UI (트림 슬라이더)
4. **Save 버튼** → 트림 저장 API

### Phase 4: 쇼츠 생성 & 재생
1. **쇼츠 생성 API** (`POST /api/v1/shorts/generate`)
2. **쇼츠 상태 폴링 API** (`GET /api/v1/shorts/status`)
3. **쇼츠 저장 API**
4. **ShortsResultModal** 체크박스 선택 + 저장 연동
5. **ShortsPlaybackModal** 신규 생성 (재생/일시정지)
6. **쇼츠 클릭 시** 상세 + 재생 모달 동시 표시

### Phase 5: 부가 기능
1. **댓글 타임라인** (댓글 데이터 API + 차트)
2. **원본/쇼츠 클릭 분기** (쇼츠 → 재생 모달)
3. **알림 연동** (실제 백엔드 알림 API)

## 📌 6. 용어 정리 (초보자용)

- **MSW**: Mock Service Worker – 백엔드 없이 프론트엔드에서 가짜 API 응답을 주입
- **FSD**: Feature-Sliced Design – 폴더 구조 설계 방식 (entities, features, widgets 등)
- **Zustand**: 상태 관리 라이브러리 (useShortsStore)
- **Recording**: 스튜디오에서 라이브 스트리밍 시 녹화된 영상
- **Clip/Shorts**: 원본 영상에서 AI가 추출한 하이라이트 구간 (짧은 영상)
- **API Gateway**: 모든 API 요청이 먼저 거치는 진입점, 여기서 core/media 등으로 라우팅
