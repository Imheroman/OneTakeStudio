# OneTake 프로젝트 중간 보고서 및 스튜디오 영상 송출 플로우

> 현재까지 진행된 프로젝트 전반 정리 및 스튜디오에서 영상을 플랫폼(YouTube 등)으로 송출하는 과정을 상세 기술한 문서입니다.

---

## Part 1. 프로젝트 전체 정리

### 1.1 진행 상황 요약

| 영역 | 상태 | 비고 |
|------|------|------|
| 인증/회원 | 로그인·회원가입·OAuth(Google/Kakao/Naver) | Core + API Gateway 연동 |
| 워크스페이스 | 홈·라이브러리·채널·멤버·스토리지·마이페이지 | 레이아웃·사이드바·상단바 |
| 스튜디오 | 씬·소스·프리뷰·레이아웃·배너·에셋·스타일·노트·녹화 | Konva 캔버스 기반 송출 화면 |
| 송출(Go Live) | LiveKit 연결·RTMP 송출·플랫폼 선택 | Media 서비스 + LiveKit |
| 채널 연동 | YouTube 등 목적지 연동·RTMP URL/Stream Key | Core 서비스 채널 API |
| 녹화 | 로컬(MediaRecorder)·클라우드(LiveKit Egress) | 프론트/백엔드 모두 구현 |

### 1.2 구현 정도

- **프론트엔드**: Next.js(App Router), 스튜디오 진입 ~ 캔버스 편집 ~ Go Live ~ 송출 중지까지 일련 플로우 구현. 배너·에셋·스타일 오버레이, 에셋 드래그/리사이즈, 스타일 색상 연동 포함.
- **백엔드**: MSA 구조(API Gateway, Eureka, Core Service, Media Service). 로그인/회원·스튜디오·채널·알림·스트림 토큰·송출(Publish)·녹화 등 API 구현.
- **LiveKit**: 토큰 발급(백엔드) → 프론트 WebRTC 연결 → Room 내 미디어 퍼블리시 → Egress(RTMP/녹화)는 백엔드가 LiveKit 서버에 지시.

### 1.3 사용 기술

| 구분 | 기술 |
|------|------|
| 프론트 | Next.js 14+, React, TypeScript, Zustand, TailwindCSS, react-konva, livekit-client, axios |
| 백엔드 | Java 17+, Spring Boot, Spring Cloud Gateway, Eureka, JWT |
| 미디어/송출 | LiveKit(WebRTC SFU), WebRTC, Canvas captureStream, RTMP |
| 인프라 | Docker, Eureka, MSA 라우팅 |

### 1.4 전체 플로우(사용자 관점)

1. 로그인/회원가입 → 워크스페이스 진입  
2. 스튜디오 생성/선택 → 씬·소스(카메라/화면공유/이미지 등) 추가  
3. 프리뷰 캔버스에서 레이아웃·배너·에셋·스타일 편집  
4. Go Live 클릭 → 송출할 채널(플랫폼) 선택 → 송출 시작  
5. LiveKit으로 영상 전송 → 백엔드가 RTMP 송출 지시 → YouTube 등에서 라이브 스트림 수신  
6. 송출 중지 시 RTMP 중지·LiveKit 연결 해제

### 1.5 기능 정리

- **인증**: 이메일 로그인, 회원가입, OAuth, JWT(access/refresh), 토큰 만료 처리  
- **워크스페이스**: 사이드바(Home/Library/Channels/Members/Storage), 상단바(로고·알림·로그아웃)  
- **스튜디오**: 씬 CRUD, 소스 추가/순서/토글/드래그·리사이즈, 해상도(720p/1080p), 배너(티커)·에셋(로고/오버레이/비디오)·스타일(색상/버블), 노트·멤버·녹화  
- **송출**: LiveKit Room 연결, 캔버스 스트림 퍼블리시, 플랫폼별 RTMP 송출·중지·상태 조회  
- **채널**: 연동 채널 목록·RTMP URL/Stream Key(Core 연동)

### 1.6 프론트엔드 정리

- **경로**: `(auth)`, `(landing)`, `(main)` 레이아웃. 스튜디오는 `(main)` 하위 `/studio/[id]`에서 전용 레이아웃(사이드바 없음) 적용.
- **상태**: `useAuthStore`(Zustand persist)로 사용자·accessToken·refreshToken·로그인 여부 관리. API 요청 시 `ApiAuthProvider`가 axios 인터셉터로 `Authorization: Bearer {accessToken}`, `X-User-Id` 부여.
- **스튜디오**: `useStudioMain`에서 씬·소스·LiveKit Room·송출 상태·`getPreviewStreamRef` 관리. `PreviewArea`는 Konva Stage/Layer로 캔버스 렌더링 후 `getPreviewStreamRef.current()`로 `canvas.captureStream(30)` 반환.
- **송출**: Go Live 시 `joinStream`(토큰 발급) → `room.connect(livekitUrl, token)` → 캔버스 MediaStream 비디오 트랙 + 마이크 오디오 트랙 퍼블리시 → `startPublish(studioId, destinationIds)` 호출.

### 1.7 백엔드 정리

- **API Gateway(60000)**: JWT 검증(`JwtUserIdFilter`), `Authorization: Bearer`에서 추출한 `iid`를 `X-User-Id`로 하위 서비스에 전달. `/api/streams/**` → media-service `/api/media/stream/**`, `/api/publish/**` → media-service `/api/media/publish` 재작성.
- **Core Service**: 인증·회원·스튜디오·채널(목적지)·알림·즐겨찾기 등 REST API.
- **Media Service**: 스트림 토큰(`/api/media/stream/join`), 퇴장(`leave`), 송출 시작/중지/상태(`/api/media/publish`), LiveKit 토큰 생성·Room·Egress(RTMP/녹화) 호출, Core와 채널 정보 연동.

## Part 2. 스튜디오 영상 송출 플로우 (상세)

아래는 **스튜디오에서 캔버스 영상을 만들고, 송출 시작 버튼을 눌러 플랫폼을 선택한 뒤 실제로 송출이 이루어지는** 과정을 단계별·기술별로 정리한 내용입니다.

### 2.1 1단계: 프론트에서 캔버스에 영상 만들기

- **역할**: 사용자가 보는 “송출 화면”은 **Konva(react-konva)의 Stage/Layer = 하나의 HTML Canvas**입니다.
- **구성**:
  - **소스**: 카메라(`getUserMedia`), 화면공유(`getDisplayMedia`), 이미지 등이 각각 `MediaStream`으로 얻어지고, `<video>`에 `srcObject`로 붙습니다.
  - **그리기**: Konva `Layer` 위에 각 소스를 `Image` 노드로 두고, `requestAnimationFrame`으로 매 프레임 `img.image(video)` + `layer.batchDraw()`를 호출해 비디오 프레임을 캔버스에 그립니다.
  - **레이아웃**: 소스별 위치·크기·z-order는 `sourceTransforms`와 `arrangeSourcesInLayout`으로 계산되며, 드래그·리사이즈로 변경할 수 있습니다.
  - **배너·에셋·스타일**: 같은 Layer 위에 Rect, Text, Image(에셋) 등으로 하단 배너, 티커 텍스트, 에셋 오버레이, 스타일(색상/버블)이 겹쳐져 그려집니다.
- **결과**: “송출에 쓸 한 장의 영상”은 **이 캔버스에 실시간으로 그려진 내용**입니다. 이걸 스트림으로 빼려면 다음 단계의 `captureStream`이 사용됩니다.

### 2.2 2단계: 송출 시작 버튼·플랫폼 선택·송출 시작

- **UI**: 스튜디오 상단 등에서 **Go Live** 버튼을 누르면 `GoLiveConfirmModal`이 열립니다.
- **플랫폼 선택**: 연동된 채널(목적지) 목록이 표시되고, 사용자가 송출할 채널(YouTube 등)을 선택한 뒤 **Go Live**(또는 “송출 시작”) 버튼을 누릅니다.
- **프론트 처리**: `handleGoLive(selectedDestinationIds)`가 호출됩니다.  
  - 선택된 채널 ID 배열 `destinationIds`가 그대로 백엔드 `startPublish`에 전달됩니다.

### 2.3 3단계: 프론트 → LiveKit으로 영상 전송

- **순서**:
  1. **토큰 발급**: `joinStream({ studioId, participantName })` 호출.  
     - 프론트는 **로그인 시점에 받은 JWT(accessToken)**를 `Authorization: Bearer`로, 사용자 식별자를 `X-User-Id`로 보냅니다.  
     - API Gateway가 JWT를 검증하고 `X-User-Id`를 붙여 Media 서비스로 전달합니다.  
     - Media 서비스는 **이미 로그인된 사용자(토큰 검증 통과)**라고 보고, LiveKit **Room 접속용 JWT**를 생성해 반환합니다(livekitUrl, token, roomName 등).
  2. **LiveKit Room 연결**: 프론트는 `livekit-client`의 `Room`으로 `room.connect(livekitUrl, token)`를 호출합니다.  
     - LiveKit 서버는 **이 토큰이 우리 백엔드가 서명한 것인지** 검증하고, 통과하면 WebRTC 연결을 수립합니다.
  3. **캔버스 스트림 퍼블리시**:  
     - `getPreviewStreamRef.current()`로 **Konva Layer의 Canvas**에서 `canvas.captureStream(30)`을 호출해 **MediaStream**을 얻습니다.  
     - 이 MediaStream의 **비디오 트랙**을 `room.localParticipant.publishTrack(videoTrack, { name: "canvas", source: Camera })`로 LiveKit Room에 퍼블리시합니다.  
     - 마이크는 별도로 `createLocalTracks({ video: false, audio: true })` 후 같은 식으로 퍼블리시합니다.
- **정리**: “캔버스에 그려진 영상”은 **브라우저 내에서만** 존재하고, **영상 파일이 서버 디스크에 저장되는 것이 아닙니다**.  
  브라우저가 **실시간 MediaStream(WebRTC 트랙)**으로 LiveKit 서버로 보내면, LiveKit은 그 스트림을 **Room 안에서 메모리/버퍼 단위로 다루며**, 필요 시 Egress(송출/녹화)로 내보냅니다.

### 2.4 4단계: 백엔드에서 로그인 정보를 “다시” 확인하는 이유와 방식

송출·스트림 참가 같은 중요한 동작에서는 **요청이 올 때마다** “지금 이 요청을 보낸 사람이 누구인지, 로그인된 사용자인지”를 서버가 다시 확인합니다.  
그래서 “로그인 정보를 다시 확인한다”고 표현한 것입니다.

#### 2.4.1 로그인 정보는 어디에 저장되나?

- **프론트**:
  - **accessToken(JWT)**: Zustand `useAuthStore`에 보관되고, `persist` 미들웨어로 **localStorage** 키 `onetake-auth`에 직렬화되어 저장됩니다. (새로고침 후에도 유지)
  - **refreshToken**: 로그인 응답 시 별도로 **localStorage** `refreshToken`에 저장됩니다.
  - **user**: `userId`, `email`, `nickname`, `profileImageUrl` 등이 `onetake-auth` 안에 함께 persist 됩니다.
- **백엔드**:
  - 로그인 “세션”을 서버 메모리/DB에 두지 않고, **무상태(stateless)** 방식입니다.  
  - 매 요청마다 클라이언트가 보낸 **JWT(accessToken)**만 검증해서 “이 사용자”를 식별합니다.  
  - 따라서 “로그인 정보”는 사실상 **JWT 안에 들어 있는 클레임(sub, iid, email 등)**이고, 서버는 이 토큰이 **서명이 맞는지·만료되지 않았는지**만 검사합니다.

#### 2.4.2 로그인 정보는 어떤 식으로 관리되나?

- **발급**: 로그인/회원가입 성공 시 Core 서비스가 JWT를 생성해 accessToken(및 refreshToken)을 내려줍니다.  
  JWT에는 보통 `sub`(주로 UUID), `iid`(내부 숫자 ID), `email` 등이 들어갑니다.
- **사용**: 프론트는 **모든 API 요청**에 `Authorization: Bearer {accessToken}`을 붙입니다.  
  `ApiAuthProvider`에서 axios 인터셉터가 `useAuthStore.getState().accessToken`을 꺼내 헤더에 세팅합니다.  
  추가로 `X-User-Id`는 프론트에서 `user.userId`로 넣을 수 있지만, **서버는 Gateway에서 JWT를 파싱해 나온 `iid`를 `X-User-Id`로 덮어써서** 최종적으로 “JWT에 담긴 사용자”를 신뢰합니다.
- **만료**: accessToken 만료 시 프론트 `checkAuth()`에서 `isTokenExpired(accessToken)`로 판단하고, 로그아웃 처리하거나 refresh로 재발급받는 플로우를 탈 수 있습니다.

#### 2.4.3 “다시 확인”하는 방식

- **API Gateway (`JwtUserIdFilter`)**  
  - `Authorization: Bearer {token}`이 있으면, **매 요청마다** JWT 서명 검증 및 파싱을 수행합니다.  
  - 검증 성공 시 `claims`에서 `iid`를 꺼내 `X-User-Id` 헤더로 넣고, Media/Core 등 하위 서비스로 전달합니다.  
  - 검증 실패(만료·위조·형식 오류)면 JWT 관련 헤더를 붙이지 않고 통과시키며, 이후 Media/Core에서 `X-User-Id`가 없으면 401 등으로 거절할 수 있습니다.
- **Media 서비스 (스트림/송출)**  
  - `StreamController.joinStream`, `PublishController.startPublish` 등은 **`@RequestHeader("X-User-Id") Long userId`**로 사용자를 받습니다.  
  - 이 `X-User-Id`는 **Gateway가 JWT에서 채워 준 값**이므로, “이 요청은 JWT 검증을 통과한 로그인 사용자의 요청이다”를 전제로 합니다.  
  - 즉, **토큰 발급·송출 시작 시점에 “로그인 정보를 다시 확인”하는 것은, Gateway에서 JWT를 다시 검증하고 userId를 다시 넣는 것**으로 이루어집니다.

#### 2.4.4 다시 확인하는 이유 (이론적 보충)

- **보안**: 한 번 로그인했다고 해서 브라우저가 “로그인됨” 상태만 갖고 있고, 서버가 아무 검증 없이 요청을 받으면 악의적인 요청(토큰 도난·재사용)을 막기 어렵습니다.  
  그래서 **중요한 API(토큰 발급, 송출 시작)**는 반드시 **서버가 현재 요청에 붙은 JWT를 검증**해 “지금 이 사용자가 누구인지”를 다시 확인합니다.
- **무상태**: 서버가 “현재 로그인한 사용자 목록”을 세션 저장소에 두지 않으므로, **매 요청의 JWT 검증**이 “로그인 여부 확인”의 유일한 수단입니다.
- **권한·감사**: “누가 언제 송출을 시작했는지”를 남기려면, 요청 시점의 사용자 ID가 필요하고, 이는 **그 요청에 포함된 JWT를 검증한 결과**에서만 신뢰할 수 있습니다.

정리하면, **로그인 정보는 프론트 localStorage의 JWT(및 user)로 보관되고, “다시 확인”은 API Gateway가 매 요청마다 JWT를 검증해 `X-User-Id`를 채우는 방식**으로 이루어지며, **보안·무상태·권한/감사**를 위해 의도적으로 그렇게 설계된 것입니다.

### 2.5 5단계: LiveKit이 “로그인 확인 신호”를 받으면 플랫폼으로 송출

- **실제 구현 관계**:  
  LiveKit 서버는 **“백엔드가 로그인을 확인했다”는 별도 신호를 받지는 않습니다**.  
  대신 **프론트는 이미 “로그인 확인이 끝난 뒤”**에만 토큰을 받을 수 있고, 그 토큰으로만 LiveKit에 접속할 수 있습니다.  
  즉, “로그인 확인”과 “LiveKit 연결/송출”은 **시간 순서**로만 연결되어 있습니다.
- **플로우**:
  1. 프론트: `joinStream` 호출 → **Gateway가 JWT 검증** → Media가 **LiveKit Room 접속용 토큰** 발급. (이때 “로그인 확인” 완료)
  2. 프론트: 발급받은 토큰으로 LiveKit Room 접속 후 **캔버스 비디오 + 오디오 트랙 퍼블리시**. (영상이 LiveKit 서버로 실시간 전송)
  3. 프론트: `startPublish({ studioId, destinationIds })` 호출.  
     - 다시 **JWT 검증** 후 Media의 `PublishController.startPublish`가 호출되고, `userId`는 `X-User-Id`로 전달됩니다.
  4. Media 서비스 (`PublishService.startPublish`):  
     - 해당 스튜디오의 **활성 스트림 세션**(LiveKit Room에 대응) 존재 여부 확인.  
     - LiveKit Room에 **참가자가 있는지** `liveKitService.listParticipants(roomName)`로 확인. (여기서 “프론트가 이미 Room에 들어와 있다” = 영상이 LiveKit에 올라와 있음)  
     - Core에서 선택된 채널(destinationIds)의 **RTMP URL + Stream Key** 조회.  
     - **LiveKit Egress**를 사용해 “해당 Room의 합성 영상을 **RTMP**로 지정된 URL(YouTube 등)에 푸시”하도록 지시합니다.
  5. LiveKit 서버: Room에 들어온 트랙(캔버스 비디오 + 오디오)을 **합성(Composite)**한 뒤, **RTMP 스트림**으로 YouTube 등 플랫폼에 전송합니다.  
     → 플랫폼에서 “라이브 스트림”으로 수신·재생됩니다.

즉, **“로그인 확인 신호가 들어오면 LiveKit이 플랫폼으로 보낸다”**기보다는,  
**“로그인 확인이 된 사용자만 토큰을 받고, 그 토큰으로 LiveKit에 올리고, 그 다음 별도 API(startPublish)로 백엔드가 LiveKit에 ‘이 Room을 RTMP로 내보내라’고 지시한다”**가 정확한 순서입니다.

## Part 3. LiveKit 동작 위치·관리 주체·데이터 흐름

### 3.1 LiveKit이 동작하는 위치

- **LiveKit 서버**는 **별도 프로세스(또는 Docker 컨테이너)**로 동작하는 **미디어 서버**입니다.  
  프로젝트에서는 `livekit.yaml`, `docker-compose` 등으로 **우리 인프라(또는 동일 네트워크)** 안에서 실행되는 것으로 가정됩니다.
- **역할**: WebRTC **SFU(Selective Forwarding Unit)**.  
  - 브라우저(프론트)와 **WebRTC**로 연결되어 **영상/음성 트랙**을 받고,  
  - Room 단위로 참가자 트랙을 관리하며,  
  - **Egress** 기능으로 “Room 합성 영상”을 **RTMP 스트림** 또는 **파일 녹화**로 내보냅니다.
- **위치 요약**:  
  - **네트워크상**: 프론트와 같은 내부/공개망에서 접근 가능한 **별도 호스트(예: 동일 서버 또는 전용 미디어 서버)**.  
  - **아키텍처상**: **우리 백엔드(Media 서비스)와는 분리**되어 있고, Media 서비스는 **LiveKit API(토큰 생성·Room·Egress)**를 **클라이언트 라이브러리**로 호출해 “지시”만 합니다.  
  - 실제 미디어 처리(수신·인코딩·RTMP 푸시)는 **LiveKit 서버**가 수행합니다.

### 3.2 LiveKit의 관리 주체

- **운영 주체**: OneTake 프로젝트 팀이 **LiveKit 서버를 설치·설정·운영**합니다.  
  (오픈소스 LiveKit 서버를 쓰거나, LiveKit Cloud를 쓰면 해당 서비스 정책에 따릅니다.)
- **애플리케이션 관점**:  
  - **토큰 발급·Room 이름·Egress 시작/중지**는 **우리 백엔드(Media 서비스)**가 **LiveKit API 키/시크릿**으로 제어합니다.  
  - **미디어 스트림의 수신·혼합·인코딩·RTMP 전송**은 **LiveKit 서버**가 수행하므로, “미디어 처리”의 실행 주체는 LiveKit입니다.

### 3.3 프론트에서 영상을 보내면, 영상 소스가 “어디로” 가고 “임시 저장”되는지

- **영상 소스 자체**는 **파일로 저장되지 않습니다**.  
  - 프론트: Konva 캔버스 → `canvas.captureStream(30)` → **MediaStream**.  
  - 이 MediaStream은 **WebRTC**를 통해 **LiveKit 서버**로 **실시간 전송**됩니다.
- **LiveKit 서버 측**:  
  - 수신한 트랙은 **메모리/버퍼**에서 디코딩·합성·재인코딩됩니다.  
  - “임시 저장”은 **스트리밍용 버퍼** 수준이며, **Egress(녹화)**를 켜면 지정한 경로에 **파일로 저장**할 수 있습니다.  
  - **RTMP 송출만** 할 때는, 영상을 **파일로 남기지 않고** 실시간으로 인코딩해 RTMP URL로 푸시합니다.
- **정리**:  
  - **출발**: 브라우저 캔버스 → **실시간 MediaStream(WebRTC)**  
  - **중간**: LiveKit 서버 **메모리/버퍼** (파일 임시 저장은 Egress 녹화 시에만 해당)  
  - **도착**: RTMP로 플랫폼(YouTube 등)에 실시간 전달.

### 3.4 플랫폼까지 이동하는 방법

- LiveKit **Egress** 중 **Stream Egress(RTMP)**를 사용합니다.  
  - Media 서비스가 `LiveKitEgressService.startRtmpStream(roomName, rtmpUrls, videoQuality)`를 호출합니다.  
  - LiveKit 서버는 해당 **Room**의 합성 영상(비디오+오디오)을 **RTMP 프로토콜**로 인코딩해,  
    Core에서 조회한 **RTMP URL + Stream Key**(YouTube 등이 제공)로 **푸시**합니다.  
- YouTube 등 플랫폼은 **RTMP 인제스트**를 받아서, 자신들의 라이브 스트리밍 파이프라인으로 넣고 시청자에게 배포합니다.

### 3.5 출발부터 플랫폼 도착까지 데이터 흐름 요약

```
[1] 브라우저 (프론트)
    - Konva Layer Canvas
    - canvas.captureStream(30) → MediaStream (비디오 트랙)
    - 마이크 → 오디오 트랙
    ↓
[2] API 호출 (모두 Authorization: Bearer {accessToken} 포함)
    - POST /api/streams/join  → Gateway JWT 검증 → Media: LiveKit 토큰 발급
    - room.connect(livekitUrl, token)  → WebRTC 연결 수립
    - localParticipant.publishTrack(canvasVideoTrack) / publishTrack(audioTrack)
    ↓
[3] LiveKit 서버
    - WebRTC로 비디오·오디오 트랙 수신
    - Room 내에서 합성(Composite) · 버퍼링 (파일 저장 없음, RTMP만 할 때)
    ↓
[4] 프론트: POST /api/publish  (studioId, destinationIds)
    - Gateway JWT 검증 → X-User-Id 전달 → Media PublishController
    ↓
[5] Media 서비스
    - 활성 스트림 세션·LiveKit Room 참가자 확인
    - Core에서 destinationIds에 해당하는 RTMP URL + Stream Key 조회
    - LiveKit Egress API: “이 Room을 RTMP URL로 푸시하라” 지시
    ↓
[6] LiveKit 서버
    - Room 합성 영상을 RTMP로 인코딩
    - YouTube 등이 제공한 rtmp://.../live2/xxxx + streamKey 로 푸시
    ↓
[7] 플랫폼 (YouTube 등)
    - RTMP 인제스트 수신 → 라이브 스트림으로 변환 → 시청자에게 배포
```

- **로그인 확인**은 [2]와 [4]에서 **API Gateway가 JWT를 검증**하고 `X-User-Id`를 붙이는 단계에서 이루어지며,  
  LiveKit은 “로그인 확인”이 아닌 **“유효한 Room 접속 토큰”**과 **“Egress 지시”**만 받습니다.
- **영상 데이터**는 [1]→[2]→[3]에서 **WebRTC**로, [3]→[6]→[7]에서 **RTMP**로 이동하며,  
  **파일로 임시 저장되는 구간은 Egress 녹화를 켰을 때만** LiveKit이 지정한 경로에 해당합니다.
