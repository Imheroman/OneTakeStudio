# Schema 구조 방식 비교 분석

## 📋 현재 프로젝트 상황

### 프로젝트 규모
- **엔티티 수**: 7개 (Studio, User, Channel, Video, Favorite, Notification, Storage)
- **스키마 수**: 약 30개
- **API 엔드포인트**: 약 21개
- **팀 규모**: 중소규모 (추정)
- **아키텍처**: FSD (Feature-Sliced Design)

### 현재 구조
```
entities/
  studio/
    model/
      schemas.ts  (zod 스키마 + 타입 추론)
      index.ts     (스키마와 타입 export)
```

**참고**: `types.ts` 파일은 YAGNI 원칙에 따라 완전히 제거되었습니다.

---

## 🔍 두 가지 방식 비교

### 방식 1: 현재 방식 (통합형)

**구조:**
```
schemas.ts: z.object 정의 + z.infer 타입 추론
index.ts:   스키마와 타입 모두 export
```

**장점:**
- ✅ **단순함**: 한 파일에 모든 것이 있음
- ✅ **유지보수 용이**: 스키마와 타입이 같은 곳에 있어 수정이 쉬움
- ✅ **중소규모 프로젝트에 적합**: 파일 수가 적어 관리가 쉬움
- ✅ **실수 방지**: 스키마와 타입이 분리되지 않아 동기화 문제 없음
- ✅ **빠른 개발**: 스키마 정의 후 바로 타입 사용 가능

**단점:**
- ⚠️ 파일이 커질 수 있음 (하지만 현재는 문제 없음)
- ⚠️ 스키마만 필요한 경우와 타입만 필요한 경우 구분이 어려움

**코드 예시:**
```typescript
// schemas.ts
export const StudioSchema = z.object({ ... });
export type Studio = z.infer<typeof StudioSchema>; // 바로 아래에 정의
```

---

### 방식 2: 제안된 방식 (분리형)

**구조:**
```
schemas.ts: z.object 정의만
types.ts:   z.infer를 통한 타입 정의만
index.ts:   선택적 export
```

**장점:**
- ✅ **명확한 책임 분리**: 스키마와 타입이 분리됨
- ✅ **대규모 프로젝트에 적합**: 파일이 많을 때 구조가 명확함
- ✅ **선택적 import**: 스키마만 필요할 때, 타입만 필요할 때 구분 가능
- ✅ **확장성**: 나중에 스키마가 복잡해져도 관리하기 쉬움

**단점:**
- ❌ **파일 수 증가**: 엔티티당 3개 파일 (현재는 2개)
- ❌ **동기화 필요**: schemas.ts 수정 시 types.ts도 수동 업데이트 필요
- ❌ **복잡도 증가**: 중소규모 프로젝트에서는 오버엔지니어링일 수 있음
- ❌ **실수 가능성**: types.ts에서 z.infer를 깜빡할 수 있음

**코드 예시:**
```typescript
// schemas.ts
export const StudioSchema = z.object({ ... });

// types.ts
import { StudioSchema } from "./schemas";
export type Studio = z.infer<typeof StudioSchema>;
```

## 🎯 현재 프로젝트에 적합한 방식

### 추천: **방식 1 (현재 방식 - 통합형)**

### 이유:

#### 1. 프로젝트 규모
- **엔티티 7개, 스키마 30개**: 중소규모
- 분리형의 이점이 크지 않음
- 오히려 파일 수 증가로 복잡도만 올라감

#### 2. 개발 속도
- **빠른 개발**: 스키마 정의 후 바로 타입 사용
- **실수 방지**: 스키마와 타입이 같은 곳에 있어 동기화 문제 없음
- **간단한 구조**: 팀원들이 이해하기 쉬움

#### 3. 유지보수성
- **한 곳에서 관리**: 스키마 수정 시 타입도 자동 반영
- **FSD 구조와 조화**: entities 레이어가 단순하게 유지됨

#### 4. 실용성
- **YAGNI 원칙**: "You Aren't Gonna Need It"
- 현재 규모에서는 분리형의 이점이 크지 않음
- 필요할 때 리팩토링 가능
