# 장면 편집·전환 권한 개선 분석

## 1. 요구사항 정리

### 1.1 Add Scene 플로우

- 장면 추가 모달에서 이름 입력 후 저장 → **생성 직후 해당 장면으로 자동 전환**하여 소스 추가·편집 가능

### 1.2 호스트 관점

| 상황           | 권한                                                                               |
| -------------- | ---------------------------------------------------------------------------------- |
| 송출 중인 장면 | 편집 권한(락) 없어도 **편집 가능**                                                 |
| 비송출 장면    | 기존처럼 락 필요                                                                   |
| 장면 전환      | 매니저가 수정 중인 장면으로 전환 시 → **최종 저장본으로 전환**, 매니저 편집권 상실 |

### 1.3 매니저 관점

| 상황             | 권한                                        |
| ---------------- | ------------------------------------------- |
| 송출 중인 장면   | 편집 권한 있어도 **편집 불가**              |
| 비송출 장면      | 락 획득 시 편집 가능                        |
| 장면 리스트 클릭 | **편집할 장면 선택** (프리뷰에 표시) ≠ 추천 |
| 장면 추천        | **별도 추천 버튼**으로 구분                 |

### 1.4 매니저 임시 저장

- 호스트가 매니저가 수정 중이던 장면으로 전환 → 매니저 저장 시 **임시 저장**
- 호스트가 해당 장면을 이탈하면 → 임시 저장본 **적용·저장** 가능

### 1.5 UI

- **송출 중인 장면**: 장면 리스트에서 **빨간색 테두리**로 표시

## 2. 현재 구조 vs 변경 필요 사항

### 2.1 현재 구조

| 항목             | 현재                                        | 비고                       |
| ---------------- | ------------------------------------------- | -------------------------- |
| activeSceneId    | 단일 값 (송출/편집 겸용)                    | 호스트·매니저 동일         |
| canEdit          | `isEditMode && hasLock && !isLockedByOther` | 장면별 분기 없음           |
| canSwitchScene   | isHost (호스트만)                           | 호스트만 장면 클릭 시 전환 |
| 매니저 장면 클릭 | 아무 동작 없음                              | canSwitchScene=false       |
| 편집 락          | 스튜디오 전체 1개                           | 장면별 락 없음             |

### 2.2 변경 필요

| 항목                 | 변경 내용                                                         |
| -------------------- | ----------------------------------------------------------------- |
| **previewSceneId**   | 프리뷰에 표시할 장면 (편집 대상)                                  |
| **broadcastSceneId** | 호스트가 송출 중인 장면                                           |
| 호스트               | previewSceneId = broadcastSceneId                                 |
| 매니저               | previewSceneId = 클릭한 장면, broadcastSceneId = 상태 동기화 수신 |
| canEdit              | 호스트+송출장면 → 항상 true / 매니저+송출장면 → 항상 false        |
| 장면 클릭            | 호스트: 전환 / 매니저: 편집 대상 선택                             |
| 추천 버튼            | 장면 클릭과 분리, 별도 UI                                         |

## 3. 잠재적 충돌·꼬임 지점

### 3.1 편집 락 vs 호스트 송출 장면 편집

**충돌**: 호스트는 송출 장면을 락 없이 편집 가능해야 함.

**현재**: `canEdit = isEditMode && hasLock && !isLockedByOther` → 락 없으면 편집 불가

**해결**: canEdit 계산 시 예외 추가

```
canEdit =
  (호스트 && 송출중 && previewSceneId === broadcastSceneId)
  || (락보유 && 편집모드 && !다른사람락)
  && !(매니저 && 송출중 && previewSceneId === broadcastSceneId)
```

즉, **호스트 + 송출 중인 장면 편집**이면 락 무시.

---

### 3.2 매니저가 송출 장면 클릭 시

**시나리오**: 매니저가 송출 중인 장면을 클릭해 편집하려 함.

**요구**: 송출 중인 장면은 매니저가 편집 불가.

**해결**:

- 매니저가 송출 장면 클릭 → previewSceneId는 설정되지만 canEdit=false
- 프리뷰에는 표시되나 **읽기 전용** (또는 "송출 중인 장면은 편집할 수 없습니다" 안내)
- 또는 송출 장면 클릭 시 **클릭 자체를 막거나** 안내 표시

---

### 3.3 호스트 장면 전환 시 매니저 편집권 상실

**시나리오**: 매니저가 A장면 편집 중 → 호스트가 A장면으로 전환.

**요구**:

1. 호스트 화면: A장면 최종 저장본으로 전환
2. 매니저: A장면 편집권 상실
3. 매니저가 저장 시 → **임시 저장**

**충돌**:

- 현재 락은 **스튜디오 단위** → 매니저가 A장면만 편집 중이어도 락은 전체 스튜디오
- 호스트 전환 시 매니저 락을 **강제 해제**할지, **장면별로만 무효화**할지 결정 필요

**옵션 A**: 호스트 전환 시 매니저 락 강제 해제

- 장면 A뿐 아니라 다른 장면 편집도 중단
- 매니저가 B장면 편집 중이었어도 락 상실

**옵션 B**: 장면별 편집 상태 (백엔드 확장)

- `lockedSceneId` 등 장면별 락
- 호스트가 A장면 전환 시 A장면 락만 해제
- 구현 복잡도 증가

**권장**: 우선 **옵션 A**로 단순화. 호스트가 장면 전환 시, 해당 장면을 편집 중이던 매니저의 락을 해제(또는 해당 장면에 대한 편집 무효화).  
다른 장면 편집 중이던 매니저는 영향 없음 → 이를 위해 **장면별 락**이 필요.  
따라서 **옵션 B(장면별 락)**가 요구사항에 더 부합.

---

### 3.4 임시 저장 (백엔드 지원 필요)

**요구**: 매니저가 편집권 상실 후 저장 시 → 임시 저장 → 호스트 이탈 시 적용.

**현재**: 씬 저장 API는 곧바로 DB 반영. 임시 저장 개념 없음.

**필요**:

1. **백엔드**: 장면별 `pendingLayout` 또는 `draftLayout` 필드
2. 또는 **프론트만**: localStorage/sessionStorage에 임시 저장, 호스트 이탈 시 API 호출

**충돌**: 백엔드 스키마·API 변경 없이 구현하려면 프론트 임시 저장만 가능.  
호스트 이탈 감지는 **상태 동기화(SCENE_SELECTED)**로 가능.

---

### 3.5 previewSceneId vs broadcastSceneId 동기화

**호스트**:

- 장면 클릭 → broadcastSceneId 변경 + broadcast
- previewSceneId = broadcastSceneId (항상 동일)

**매니저**:

- broadcastSceneId: SCENE_SELECTED 수신으로 갱신
- previewSceneId: 장면 클릭으로 설정 (호스트 전환과 무관)

**꼬임 가능성**:

- 매니저가 A장면 편집 중
- 호스트가 B→A 전환
- 매니저 화면: broadcastSceneId=A, previewSceneId=A
- 이때 매니저는 A장면 편집권 상실
- previewSceneId를 유지할지, broadcastSceneId에 맞출지?

**제안**:

- 매니저가 편집 중이던 장면으로 호스트가 전환하면 → 해당 장면에 대한 편집권 상실
- previewSceneId는 그대로 두되, **canEdit=false**로 전환
- "호스트가 이 장면을 송출 중입니다. 임시 저장만 가능합니다" 안내

---

### 3.6 Add Scene 직후 전환

**현재**: `handleAddScene` → API 호출 → `fetchStudio()` → 새 장면이 목록에 추가되지만 activeSceneId는 그대로.

**필요**: 생성 응답에서 sceneId 추출 → activeSceneId 설정.

**충돌**:

- `fetchStudio()`가 비동기 → 새 장면이 scenes에 반영된 뒤 activeSceneId 설정 필요
- API 응답에 `data.sceneId` 포함 여부 확인 필요

**해결**: create scene API 응답에서 `data.sceneId` 사용. `fetchStudio()` 후 `setActiveSceneId(String(data.sceneId))` 또는 응답 직후 설정.

---

### 3.7 장면 전환이 안 되는 문제 (추정 원인)

**가능 원인**:

1. `handleSceneSelect`가 `isStateSyncConnected`에 의존 → 오프라인이면 broadcast만 스킵, setActiveSceneId는 호출됨
2. `canSwitchScene`이 false로 전달되는 경우 (버그)
3. `editingSceneId`가 설정된 상태에서 클릭 시 `!editingSceneId` 조건으로 인해 클릭 무시
4. `studio?.scenes`와 `activeSceneId` 포맷 불일치 (예: number vs string)

**확인 필요**: StudioMain에서 `canSwitchScene={isHost}` 전달, handleSceneSelect 의존성 배열.

## 4. 구현 우선순위 제안

### Phase 1 (즉시)

1. **Add Scene 직후 해당 장면으로 전환** (API 응답 sceneId 사용)
2. **송출 중 장면 빨간 테두리** 표시
3. **매니저 장면 클릭 = 편집 대상 선택** (previewSceneId 분리)
4. **추천 버튼**을 장면 클릭과 분리 (별도 버튼)

### Phase 2 (권한 로직)

5. **호스트**: 송출 중인 장면은 락 없이 편집 가능
6. **매니저**: 송출 중인 장면은 편집 불가

### Phase 3 (고급)

7. **장면별 락** 또는 호스트 전환 시 해당 장면 편집권 무효화
8. **임시 저장** (백엔드 확장 또는 프론트 임시 저장)

## 5. 설정 부딪힘 요약

| #   | 항목                   | 충돌 내용                  | 해결 방향                                    |
| --- | ---------------------- | -------------------------- | -------------------------------------------- |
| 1   | 락 vs 호스트 송출 편집 | 락 없이 편집 필요          | canEdit 예외 로직                            |
| 2   | 매니저 송출 장면       | 편집 불가                  | canEdit 예외 + UI 안내                       |
| 3   | 호스트 전환 시 매니저  | 편집권 상실                | 장면별 락 또는 전환 시 해당 장면 무효화      |
| 4   | 임시 저장              | 현재 API 없음              | 백엔드 확장 또는 프론트 임시 저장            |
| 5   | preview/broadcast 분리 | 매니저 편집 vs 호스트 송출 | previewSceneId, broadcastSceneId 분리        |
| 6   | 장면 전환 안 됨        | 원인 미확인                | handleSceneSelect, canSwitchScene, 포맷 검증 |

---

## 6. 결론

- **Phase 1**은 기존 구조만으로 대부분 구현 가능.
- **Phase 2**는 canEdit 조건 확장으로 처리 가능.
- **Phase 3**은 **장면별 락**과 **임시 저장**을 위해 백엔드 설계가 필요하며, 우선 Phase 1·2를 적용한 뒤 요구사항을 재검토하는 것이 좋음.
