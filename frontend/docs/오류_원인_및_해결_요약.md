# Merge 후 발생 오류 원인 및 해결 요약

## 1. MSW 사용 시 404 오류

### 1-1. `GET /api/notifications 404 (Not Found)`

**원인**

- **API 요청이 MSW가 아닌 다른 origin으로 나가는 경우**: `NEXT_PUBLIC_API_URL`이 `http://localhost:60000` 등으로 설정되어 있으면, axios 요청은 **API 서버(60000)** 로 나갑니다. MSW는 **브라우저에서 현재 페이지와 같은 origin(예: localhost:3000)** 으로 나가는 요청만 가로챕니다. 따라서 60000으로 나가는 요청은 MSW가 처리하지 못하고, 실제 백엔드/게이트웨이로 가서 404 또는 CORS가 발생합니다.
- **같은 origin으로 요청하는 경우**: `NEXT_PUBLIC_API_URL`을 비워 두면 요청이 `http://localhost:3000/api/notifications`로 가고, 이때는 MSW 핸들러(`/api/notifications`)가 정상적으로 가로채야 합니다. 이렇게 했는데도 404가 나온다면 MSW 워커가 등록되지 않았거나, 빌드/캐시 문제일 수 있습니다.

**해결**

- **MSW로 목업만 쓰고 싶을 때**: `.env.local`에서 `NEXT_PUBLIC_API_URL`을 **비우거나 제거**해서, API 요청이 프론트와 같은 origin(예: `http://localhost:3000`)으로 가게 하세요. 그러면 MSW가 `/api/notifications` 등을 가로챕니다.
- **실제 백엔드(60000)를 쓰고 싶을 때**: `NEXT_PUBLIC_API_URL=http://localhost:60000`으로 두고, 백엔드/API Gateway에서 해당 경로(`/api/notifications` 등)가 구현·라우팅되어 있고 CORS도 허용되어 있어야 합니다.

### 1-2. `GET /api/workspace/undefined/studios/recent 404 (Not Found)`

**원인**

- User 엔티티에는 **`userId`** 만 있고 **`id`** 필드는 없습니다. 그런데 랜딩·탑네비 등에서 `user.id`를 사용해 `/workspace/${user.id}`로 이동하고 있어, `user.id`가 `undefined`가 되고, 결과적으로 `/workspace/undefined`로 이동했습니다.
- 워크스페이스 페이지는 URL의 `[id]`를 그대로 `userId`로 넘기므로, `userId === "undefined"`인 상태로 `/api/workspace/undefined/studios/recent`가 호출되어 404가 발생했습니다.

**해결 (프론트엔드 수정 완료)**

- **TopNav**: `user?.id` → `user?.userId`, 링크를 `/workspace/${user.userId}`로 수정.
- **Landing**: 리다이렉트를 `user.id` → `user.userId`로 수정하여 `/workspace/${user.userId}`로 이동하도록 변경.
- **WorkspaceHome**: `userId`가 없거나 문자열 `"undefined"`일 때는 최근 스튜디오 API를 호출하지 않도록 가드 추가.

이제 로그인 후 워크스페이스로 들어갈 때 올바른 `userId`가 URL과 API 요청에 사용됩니다.

## 2. MSW를 사용하지 않을 때 CORS / Network 오류

### `Access to XMLHttpRequest at 'http://localhost:60000/api/library/videos' from origin 'http://localhost:3000' has been blocked by CORS policy`

**원인**

- 브라우저는 **다른 origin**(여기서는 `http://localhost:3000` → `http://localhost:60000`)으로 요청할 때 CORS 정책을 적용합니다.
- API 서버(또는 API Gateway, 60000)가 `Access-Control-Allow-Origin` 등 CORS 헤더를 보내지 않거나, `http://localhost:3000`을 허용하지 않으면 브라우저가 응답을 막고 "blocked by CORS policy" 및 Network Error가 발생합니다.

**해결 (백엔드/API Gateway 측)**

- **API Gateway(60000)** 또는 실제 API를 제공하는 서버에서, **preflight(OPTIONS)** 와 **실제 요청(GET/POST 등)** 에 대해 다음을 허용하도록 CORS를 설정해야 합니다.
  - `Access-Control-Allow-Origin: http://localhost:3000` (또는 개발용으로 `*` — 보안상 프로덕션에서는 구체적인 origin 권장)
  - 필요 시 `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers` 등도 설정

프론트엔드만으로는 CORS를 해결할 수 없으며, **백엔드/API Gateway에서 localhost:3000 origin을 허용**하는 설정이 필요합니다.
