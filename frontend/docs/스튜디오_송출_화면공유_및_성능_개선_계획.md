# 스튜디오 송출: 화면공유 미송출·프레임 정지 원인 및 점진적 개선 계획

> 백엔드 연동 이후 **화면공유가 송출에 안 나오는 문제**와 **프레임이 거의 멈춘 수준으로 느려지는 문제**의 원인·가능성을 정리하고, **프론트 위주**로 점진적 개선하는 계획입니다.

---

## 1. 현상 요약

| 현상 | 설명 |
|------|------|
| **화면공유 미송출** | Go Live 후 유튜브 등에는 웹캠/캔버스는 나오는데, 화면공유 영역이 안 나오거나 검은 화면 |
| **프레임 거의 멈춤** | 송출 시작 후 프리뷰/송출 화면이 극심하게 끊기거나 거의 정지한 것처럼 보임 |
| **백엔드 연동 후 악화** | 위 두 현상이 백엔드(API·LiveKit·송출) 연동 후 심해짐 |

## 2. 화면공유 미송출 — 원인 및 가능성

### 2.1 캔버스에 화면공유가 그려지기 전에 캡처되는 경우 (타이밍)

- **흐름**: Go Live 시 `unpublish(video/screen)` → `waitForFrames(2)` → `waitForFrames(1)` → `getPreviewStreamRef.current()` → `canvas.captureStream(30)`.
- **가능성**: `unpublish` 직후에는 **캡처용 Stage**의 `sourceElements`가 아직 화면공유 `<video>`를 참조하고 있어도, **캡처 레이어가 한 프레임도 그리기 전**에 `captureStream()`이 호출되면 **이전 프레임(또는 빈 화면)** 이 잡힐 수 있음.
- **또한**: `localPublishedStreamsRef`에서 스트림을 쓰도록 했지만, **캡처 레이어는 `sourceElements`의 `HTMLVideoElement`를 직접 그리기** 때문에, 그 엘리먼트에 `srcObject`가 언제 채워지는지가 중요함. unpublish 후 `streamsMap`/`sourceElements`가 갱신되기 전에 캡처가 일어나면 화면공유가 빠진 상태로 캡처될 수 있음.

**관련 코드**: `PreviewArea` — `getPreviewStreamRef.current`가 `captureLayerRef`의 canvas를 `captureStream(30)` 함. `VideoSourceNode`(캡처용)가 `scheduleCaptureBatchDraw`로 **매 프레임** 그리도록 되어 있으나, **Go Live 직후 몇 프레임**은 아직 그리지 않았을 수 있음.

### 2.2 캡처 Stage에 화면공유 소스가 없거나 비어 있는 경우

- **가능성**: `sortedSources` / `displaySources`에 screen 소스는 있지만, `sourceElements.get(screenSource.id)`가 **null**이거나 **아직 설정 전**인 경우.
- **결과**: `VideoSourceNode` 대신 `Rect`(회색 placeholder)가 그려져서, 송출에는 검은/회색 영역만 나옴.
- **원인 후보**:
  - `useSourceStreams`에서 로컬 screen은 `localPublishedStreamsRef` 또는 `publishedTracks`에서 스트림을 넣는데, **unpublish 직후** `publishedTracks`에서 제거되고, `localPublishedStreamsRef`는 유지. 하지만 **React 상태인 `streamsMap`** 이 먼저 갱신되지 않아, `getSourceStream(screenId)`가 한 타이밍에 null을 줄 수 있음.
  - `PreviewArea`의 `setupSources`가 **비동기**라서, Go Live 시점에 캡처용 Stage의 `sourceElements`에 screen이 아직 반영되지 않았을 수 있음.

### 2.3 캡처 스트림이 “한 번만” 잡혀서 이후 갱신이 안 되는 경우

- **가능성**: `captureStream(30)`으로 얻은 `MediaStream`을 LiveKit에 publish한 뒤, **캡처용 Stage는 계속 그리지만**, 브라우저/ LiveKit 쪽에서 그 스트림의 프레임이 갱신되지 않는 것처럼 동작할 수 있음. (브라우저/구현 이슈)
- **가능성 2**: 캡처용 Layer가 **라이브 중에 한 번도 다시 그리지 않음** — 현재는 `VideoSourceNode`가 매 프레임 `scheduleCaptureBatchDraw`를 호출하므로 이론상 계속 그려져야 하나, **라이브 진입 직후**에만 한 번 그려지고 멈추는 경로가 있으면 화면공유가 첫 프레임만 나오거나 멈춤.

### 2.4 요약 (화면공유)

| # | 원인 후보 | 우선순위 | 비고 |
|---|-----------|----------|------|
| A | unpublish 후 캡처 타이밍: 캡처 레이어가 “화면공유 포함 한 프레임”을 그리기 전에 `captureStream()` 호출 | 높음 | waitForFrames만으로는 부족할 수 있음 |
| B | Go Live 시점에 캡처 Stage의 `sourceElements`에 screen이 아직 없음 (비동기/상태 지연) | 높음 | streamsMap / getSourceStream / sourceElements 동기화 |
| C | localPublishedStreamsRef는 있으나, 캡처 레이어가 참조하는 video 엘리먼트의 srcObject가 비어 있음 | 중간 | ref와 DOM 엘리먼트 연결 확인 |
| D | captureStream(30) 또는 LiveKit 쪽에서 첫 프레임만 사용되는 이슈 | 낮음 | 브라우저/라이브킷 문서 확인 |

## 3. 프레임 거의 멈춤 / 심각한 느려짐 — 원인 및 가능성

### 3.1 듀얼 Stage + N개 소스당 RAF 루프 (메인 스레드 과부하)

- **구조**: `PreviewArea`에 **표시용 Stage**와 **캡처용 Stage** 두 개가 있고, **비디오/화면 소스마다** `VideoSourceNode`가 하나씩 있음.
- **동작**: 각 `VideoSourceNode`는 `useLayoutEffect` 안에서 **requestAnimationFrame 루프**를 돌며 매 프레임:
  - `img.image(video)` 로 Konva Image에 비디오 프레임을 넣고
  - `scheduleBatchDraw()` (표시용) 또는 `scheduleCaptureBatchDraw()` (캡처용) 호출.
- **결과**:
  - 소스 2개(웹캠 + 화면공유)면 **표시용 2개 RAF 루프 + 캡처용 2개 RAF 루프** = 최대 4개 루프가 매 프레임 **각각** batchDraw를 예약.
  - `scheduleBatchDraw` / `scheduleCaptureBatchDraw`는 내부에서 또 **requestAnimationFrame**으로 한 번만 그리도록 되어 있지만, **여러 소스가 동시에 호출**하면 같은 프레임에 여러 번 draw가 스케줄될 수 있고, Konva가 두 Stage를 모두 그리면 **프레임당 그리기 횟수와 비용이 2배 이상**이 됨.

**관련 코드**: `PreviewArea.tsx` — `VideoSourceNode` (표시용/캡처용 각각), `scheduleBatchDraw` / `scheduleCaptureBatchDraw`.

### 3.2 canvas.captureStream(30) + 인코딩/전송

- **동작**: 송출 시 캡처용 Stage의 canvas에 대해 `captureStream(30)`을 호출해 30fps MediaStream을 만들고, 이를 LiveKit에 publish.
- **비용**: 캔버스 픽셀을 매 프레임 읽고, 인코딩·네트워크 전송이 이어지므로 **CPU/GPU 부하**가 큼. 특히 캡처용 Stage가 720p/1080p 고해상도이면 더 무거움.
- **레이스**: 위 3.1과 합쳐지면, **같은 메인 스레드**에서 “N개 RAF 루프 + Konva 두 Stage 그리기 + captureStream 읽기”가 겹쳐서 **프레임이 크게 떨어지거나 멈춘 것처럼** 보일 수 있음.

### 3.3 백엔드 연동으로 인한 추가 부하

- **API 호출**: Go Live 시 `joinStream`, `startPublish`, 채팅 연동 등으로 요청이 늘어남.
- **상태/리렌더**: `publishedTracks`, `sources`, `onStageSourceIds` 등이 바뀌면서 `useSourceStreams`, `useStudioMain` 등이 리렌더하고, 그에 따라 PreviewArea까지 리렌더될 수 있음.
- **LiveKit**: room 연결, 트랙 publish/unpublish, 미디어 인코딩/전송이 한꺼번에 일어나면 메인 스레드와 네트워크 모두 부하가 커짐.

### 3.4 요약 (성능)

| # | 원인 후보 | 우선순위 | 비고 |
|---|-----------|----------|------|
| P1 | 캡처용 Stage의 VideoSourceNode가 소스 수만큼 RAF 루프 + 매번 scheduleCaptureBatchDraw → 프레임당 그리기 과다 | 높음 | 단일 RAF로 묶거나 throttle |
| P2 | 표시용 + 캡처용 두 Stage를 매 프레임 모두 그리며, 캡처용까지 720p/1080p 전부 그리기 | 높음 | 송출 중에는 캡처용만 그리거나 해상도/주기 조정 |
| P3 | captureStream(30) + LiveKit 인코딩/전송이 메인 스레드와 겹침 | 중간 | 해상도/프레임레이트 하향 또는 워커 검토 |
| P4 | Go Live 직후 연쇄 상태 업데이트로 리렌더 과다 | 중간 | 불필요 리렌더 줄이기 |
| P5 | 채팅/기타 API 폴링·이벤트로 인한 추가 부하 | 낮음 | 필요 시 throttle/간격 조정 |

## 4. 점진적 개선 계획 (프론트 위주)

### Phase 1 — 화면공유 미송출 완화 (최소 변경)

**목표**: 캡처가 “화면공유가 한 번이라도 그려진 뒤”에 일어나도록 하고, 캐시/스트림 존재를 검증.

1. **캡처 전에 캡처 레이어를 명시적으로 한 번 그리기**
   - `PreviewArea`에서 `getPreviewStreamRef`에 넘기는 함수 외에, **“캡처용 레이어를 지금 한 번 그리기”** 콜백을 ref로 노출 (예: `requestCaptureDrawRef`).
   - `handleGoLive`에서 `getPreviewStreamRef.current()` 호출 **직전**에 해당 콜백을 호출한 뒤, `waitForFrames(1)` 한 번 더 주고 나서 `getPreviewStreamRef.current()` 호출.
   - 효과: 캡처용 Stage가 최소 한 프레임은 “현재 sourceElements 기준”으로 그린 뒤 캡처되므로, 화면공유가 들어갈 가능성 증가.

2. **화면공유 스트림이 캡처 시점에 유효한지 로그/가드**
   - `useSourceStreams`에서 로컬 screen에 대해 `localPublishedStreamsRef`를 쓸 때, `cached.active`와 트랙 상태를 한 번 더 검사.
   - Go Live 직전에 “캡처용 Stage에 사용되는 sourceElements에 screen이 있고, 해당 video.srcObject이 유효한지”를 개발 환경에서만 로그로 찍어 두면, 원인 B/C 구분에 도움.

3. **unpublish 후 sourceElements 갱신 대기**
   - unpublish 후 `streamsMap`이 갱신되어 `getSourceStream(screenId)`가 `localPublishedStreamsRef`에서 스트림을 주도록 되어 있어도, **PreviewArea의 sourceElements가 그 스트림으로 채워지는 시점**은 React 렌더/effect 이후. 필요하면 unpublish 직후에 **짧은 추가 대기**(예: requestAnimationFrame 1~2회 더) 후에 “캡처 준비” 콜백을 호출하도록 조정.

**산출물**:  
- `requestCaptureDrawRef` (또는 동일 역할) 추가 및 `handleGoLive`에서 사용.  
- (선택) 화면공유 스트림/엘리먼트 검증 로그.

---

### Phase 2 — 캡처용 Stage 그리기 횟수 축소 (성능)

**목표**: 캡처용 Stage에 대해 “소스 수만큼 RAF 루프”가 각각 `scheduleCaptureBatchDraw`를 부르지 않도록 해서, **프레임당 캡처용 그리기는 1회**로 근접시키기.

1. **캡처용 Stage 전용 단일 RAF 루프**
   - 캡처용 Stage를 그리는 쪽만 하나의 `requestAnimationFrame` 루프로 통합.
   - 루프 안에서:
     - 모든 비디오/화면 소스에 대해 Konva Image 등에 현재 프레임을 넣고,
     - **한 번만** `captureLayerRef.current?.batchDraw()` 호출.
   - 기존처럼 **각** `VideoSourceNode`가 `scheduleCaptureBatchDraw`를 부르지 않도록, **캡처용 Stage**에서는 “단일 RAF에서 한 번에 갱신”하는 모드로 전환 (예: props로 `useSingleRAF={true}` 또는 캡처 전용 wrapper 컴포넌트).

2. **표시용과 캡처용 그리기 분리 유지**
   - 표시용은 기존대로 소스별 RAF + `scheduleBatchDraw` 유지 가능 (사용자 체감이 큰 쪽).
   - 캡처용만 단일 RAF로 두어, 송출 품질은 유지하면서 **캡처용 Stage 때문에 생기는 메인 스레드 부하**를 줄임.

**산출물**:  
- 캡처용 Layer를 갱신하는 단일 RAF 로직 (PreviewArea 또는 전용 훅/컴포넌트).  
- 캡처용 `VideoSourceNode`는 “이미지 소스만 주입받고, 직접 scheduleCaptureBatchDraw를 호출하지 않음” 형태로 변경.

---

### Phase 3 — 송출 중 해상도/프레임레이트 조정 (선택)

**목표**: 송출 시 캡처/인코딩 부하를 줄여 프레임 드랍 완화.

1. **캡처 해상도**
   - 현재 720p/1080p 고정. 송출 시 한시적으로 720p만 쓰거나, 사용자 설정에 “송출 품질(낮음/보통/높음)”을 두고 보통=720p, 높음=1080p로 매핑.

2. **captureStream 프레임레이트**
   - `captureStream(30)` 대신 `captureStream(24)` 또는 20으로 낮춰서 부하를 조금 줄이기. (백엔드/유튜브 설정과 맞출 것.)

3. **송출 중 표시용 Stage 그리기 throttle**
   - 송출 중일 때는 표시용 Stage의 RAF를 30fps 또는 24fps로 throttle해서, 캡처용 + 표시용이 동시에 60fps로 그리지 않도록 할 수 있음. (체감 품질과 성능 타협.)

**산출물**:  
- 설정 또는 송출 플래그에 따른 캡처 해상도/프레임레이트 분기.  
- (선택) 표시용 Stage throttle.

---

### Phase 4 — 구조적 개선 (장기, 선택)

**목표**: 메인 스레드에서 캡처/인코딩 부하를 분리.

- **OffscreenCanvas + Worker**: 캡처용 그리기를 Worker로 옮기면 메인 스레드 부하는 줄어들지만, Konva가 오프스크린/워커를 지원하는지와 이식 비용이 큼.
- **동영상 인코딩**: 브라우저에서 인코딩을 워커로 옮기는 방식은 라이브러리/브라우저 API에 따라 다름. 현 단계에서는 “그리기 횟수·해상도·프레임레이트 줄이기”로 충분한지 먼저 확인 후 검토.

## 7. 화면 블랙아웃·유튜브 출력 불일치 — 우선순위 원인

| 순위 | 원인 | 설명 | 조치 |
|------|------|------|------|
| **1** | unpublish 시 `track.stop()` 호출 | Go Live 직전 unpublish(video/screen) 시 `unpublishTrack`이 `track.stop()` 호출 → MediaStreamTrack 종료 → `sourceElements`의 video 엘리먼트가 블랙. 캔버스가 그 엘리먼트를 그리면 블랙 출력. | unpublish만 하고 track은 stop하지 않기 (캔버스 그리기용 유지) |
| **2** | unpublish → canvas publish 타이밍 | unpublish 반영·Room 상태 전파 전에 canvas publish 시, Egress가 여러 트랙 합성 → 레이아웃 꼬임 | unpublish 후 대기 증가, Room 트랙 1개 확인 |
| **3** | setupSources 초기화 시 sourceElements 비움 | `sources`/`getSourceStream` 변경 시 `setSourceElements(new Map())` 즉시 실행 → 캡처 시점에 비어 있을 수 있음 | setupSources에서 클리어·재설정 순서 조정 |
| **3b** | **setupSources 재실행 → video 엘리먼트 교체** | Go Live 시 `publishedTracks` 변경 → `streamsMap` ref 변경 → `availableStreamIds` 새 배열 → setupSources 재실행 → **모든 video 엘리먼트를 새로 생성·교체** → 화면공유 "새로고침" 후 끊김 | `streamIdsKey`(ID 집합이 실제로 바뀔 때만 변경)로 setupSources 재실행 방지 |
| **4** | 프리뷰 가로세로비율 | 표시용 Stage scale/offset이 비율 유지 안 하면 프리뷰가 찌그러짐 | `displayScale`·레터박스 로직 검증 |

## 8. 듀얼 스테이지 관련 추가 이슈 (점진 탐색·조치)

| 현상 | 원인 후보 | 조치 |
|------|-----------|------|
| **스테이지에 소스 없이도 송출됨** | Go Live 시 `onStageSourceIds.length` 검사 없음 → 빈 캔버스 송출 | `handleGoLive`에서 `onStageSourceIds.length === 0`이면 에러 메시지 후 return |
| **화면공유만 미리 올려두고 라이브 시 블랙** | 캡처 시점에 screen용 video 엘리먼트가 아직 첫 프레임을 그리기 전이거나 `loadedmetadata` 미반영 | 스테이지에 screen 소스가 있으면 캡처 전 200ms 추가 대기 + `requestCaptureDrawRef` 한 번 더 호출 |
| **송출 중 편집 시 합치화면은 좌상단에 작게, 최근 것만 크게** | 송출 중 `sourceTransforms` 변경 시 캡처 레이어가 다음 RAF까지 갱신되지 않아 한 프레임 지연 또는 레이아웃 불일치 | `PreviewArea`에서 `isStreaming`일 때 `sourceTransformsKey`(또는 동일 역할) 변경 시 `scheduleCaptureBatchDraw()` 호출로 캡처 레이어 즉시 동기화 |
| **full 레이아웃 + 소스 2개 이상 시 한 소스만 크게 나옴** | `getLayoutGrid("full")`이 셀 1개만 반환 → 모든 소스가 같은 전체 영역에 겹쳐 그려짐 → 맨 위(최근) 소스만 보임 | `arrangeSourcesInLayout`에서 full + visible 2개 이상이면 split/three-grid/four-grid로 자동 분할 |
| **유튜브에 합친 화면이 작게, 다른 트랙이 크게** | unpublish 반영 전에 canvas publish → Egress가 여러 비디오 트랙을 합성해 기본 레이아웃(메인+PiP) 적용 | canvas publish 후 비디오 트랙이 1개가 될 때까지 최대 1.5초 폴링 대기 후 startPublish |
| **백스테이지(화면공유 등)가 송출됨** | Egress에 캔버스 외 비디오 트랙이 남아 있으면 해당 트랙이 메인으로 나감 | canvas publish 직후 `trackName !== "canvas"`인 비디오 트랙은 모두 unpublish 후 300ms 대기 |

## 9. 적용 이력

| 일자 | Phase | 적용 내용 |
|------|-------|-----------|
| (최초) | 1 | `requestCaptureDrawRef`: Go Live 직전 캡처용 레이어 명시적 1회 그리기 콜백 노출. `handleGoLive`에서 캡처 스트림 획득 전 `await requestCaptureDrawRef?.current?.()` 호출. |
| (최초) | 2 | 캡처용 Stage 단일 RAF: `VideoSourceNode`에 `registerExternalUpdate` 옵션 추가. 캡처용은 개별 RAF 대신 `registerCaptureUpdate`로 등록. `PreviewArea`에서 `captureUpdatersRef` + 단일 `useLayoutEffect` RAF로 매 프레임 한 번에 갱신 후 `batchDraw()` 1회. |
| (최근) | 7 | **블랙아웃 해결**: `unpublishTrack`에 `keepTrackAlive` 옵션 추가. Go Live 시 `keepTrackAlive: true`로 unpublish → track.stop() 생략 → 캔버스가 계속 화면공유/웹캠 그리기 가능. |
| (최근) | 7 | **타이밍 완화**: unpublish 후 120ms + waitForFrames(3) 대기 후 캡처. |
| (최근) | 4 | **프리뷰 비율**: `displayOffsetY` 추가 — 세로 레터박스 시에도 중앙 정렬로 가로세로비율 유지. |
| (최근) | 3 | **setupSources**: 전체 `setSourceElements(new Map())` 제거 → 제거된 소스만 삭제. 캡처 시점 sourceElements 공백 방지. |
| (최근) | Phase 3 | **캡처 fps**: `captureStream(30)` → `captureStream(24)`로 부하 완화. |
| (최근) | 3b | **화면공유 새로고침/끊김**: `streamIdsKey` 추가. setupSources가 `availableStreamIds` 배열 ref만 바뀌어도 재실행되던 문제 → ID 집합이 실제로 바뀔 때만 재실행. Go Live 시 video 엘리먼트 불필요 교체 방지. |
| (최근) | Phase 4 | **표시용 Stage 스로틀**: 송출 중(`isPublishing`)일 때 표시용 VideoSourceNode RAF를 매 2프레임마다만 갱신(30fps). 메인 스레드 부하 감소. |
| (최근) | §8 | **스테이지 소스 없이 송출 방지**: `handleGoLive`에서 `onStageSourceIds.length === 0`이면 "스테이지에 소스를 추가한 뒤 송출을 시작해주세요." 후 return. |
| (최근) | §8 | **화면공유 블랙 완화**: 스테이지에 screen 소스가 있으면 캡처 전 200ms 추가 대기 + `requestCaptureDrawRef` 한 번 더 호출. |
| (최근) | §8 | **송출 중 편집 시 캡처 동기화**: `PreviewArea`에서 `isStreaming`일 때 `sourceTransformsKey` 변경 시 `scheduleCaptureBatchDraw()` 호출로 캡처 레이어 즉시 반영. |
| (최근) | §8 | **full 레이아웃 + 소스 2개 이상**: `arrangeSourcesInLayout`에서 full이면서 visible 2개 이상이면 split(2)/three-grid(3)/four-grid(4+)로 자동 분할 → 합친 화면이 모두 보이도록. |
| (최근) | §8 | **Egress 단일 트랙 대기**: canvas publish 후 로컬 비디오 트랙이 1개가 될 때까지 최대 1.5초 폴링 후 startPublish → 유튜브에 합친 화면만 크게 나오도록. |
| (최근) | §8 | **백스테이지 송출 방지**: canvas publish 직후 `trackName !== "canvas"`인 비디오 트랙은 모두 unpublish. 캔버스(합친 화면)만 Egress로 전달되도록 함. |
| (최근) | 3-Stage | **Coordinate Normalization**: `sourceTransforms`를 0~1 정규화로 저장·동기화·씬 저장. 렌더 시에만 `toPixelTransform`으로 픽셀 변환. 창 크기와 무관하게 송출 해상도·비율 유지. |

이 문서는 **원인 파악과 계획**용이며, Phase 1·2·3·4·7·3b 및 §8 조치 적용 후 동작을 확인하면서 필요 시 추가 개선을 진행하거나 계획을 수정해 나가면 됩니다.
