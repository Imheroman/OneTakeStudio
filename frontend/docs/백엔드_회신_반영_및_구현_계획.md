## 1. 회신 기준으로 부족한 정보

문서만으로는 아래 항목이 불명확하거나 누락되어 있습니다. **가능하면 백엔드에 한 번만 확인**하면 좋고, 없으면 프론트에서 추정 경로로 먼저 구현한 뒤 404 등 시 수정하는 방식으로 진행할 수 있습니다.

| #   | 항목                            | 내용                                                                                                                                                                                                                                                                                   | 대응                                                                                                                                                      |
| --- | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **채팅 REST API Gateway 경로**  | 회신에는 `POST /api/media/chat` 로 되어 있는데, 현재 API Gateway 설정에는 `/api/v1/media/**` 만 등록되어 있음. 프론트에서 호출할 실제 URL이 `https://localhost:60000/api/v1/media/chat` 인지, `https://localhost:60000/api/media/chat` 인지(별도 라우트 추가 여부) 확인 필요.          | 우선 `NEXT_PUBLIC_API_URL + '/api/v1/media/chat'` 로 요청하고, 404 시 `/api/media/chat` 또는 백엔드에 경로 문의.                                          |
| 2   | **WebSocket(STOMP) 연결 URL**   | 채팅 실시간 수신용 구독 경로는 `STOMP: /topic/chat/{studioId}` 로 명시되어 있으나, **WebSocket base URL**(예: `ws://localhost:60000/ws`, `wss://...`)이 없음. STOMP 클라이언트 연결 시 필요한 주소.                                                                                    | 백엔드에 WebSocket 서버 URL(예: `ws://localhost:60000/ws`) 확인 요청. 없으면 채팅은 히스토리 조회 + 전송만 먼저 구현하고, 실시간 구독은 URL 확정 후 연동. |
| 3   | **멤버 초대 시 친구 목록 연동** | 멤버 초대 API는 `email` + `role` 만 받음. 친구(즐겨찾기) 목록에서 선택한 사용자를 초대할 때는 해당 사용자의 **email** 로 `POST /api/studios/{studioId}/members/invite` 를 호출하면 됨. 회신에 “1명씩 여러 번 호출”이라고 되어 있어, **userId가 아닌 email** 로 호출하는 것으로 해석함. | 프론트: 즐겨찾기 목록에서 선택한 사용자의 `email` 로 초대 API 여러 번 호출. 추가 확인 필요 시 “초대 시 userId 지원 여부”만 백엔드에 질의.                 |

그 외(공통 응답 형식, 인증, 에러 형식, CORS, 날짜 형식, Zod nullable 등)는 회신에 충분히 포함되어 있어 부족한 정보 없음.

## 2. 회신 요약과 기능별 구현 방식

| 기능               | 백엔드 상태             | 프론트 구현 방식                                                                                                                                                                                                                                   |
| ------------------ | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **채팅**           | ✅ WebSocket + REST     | STOMP 구독(`/topic/chat/{studioId}`) + `POST /api/media/chat`(또는 `/api/v1/media/chat`), 히스토리 `GET /api/media/chat/{studioId}`. WebSocket URL 확정 전에는 히스토리 + 전송만 구현 가능.                                                        |
| **배너**           | ❌ 미구현               | MSW 목업. 스키마·CRUD는 프론트에서 가정 후 패널만 구현. 추후 백엔드 API 생기면 연동.                                                                                                                                                               |
| **에셋**           | ❌ 미구현               | MSW 목업. 로고/오버레이/비디오클립 타입 가정, 목업 목록·삭제. 추후 API 연동.                                                                                                                                                                       |
| **스타일**         | ❌ 미구현               | 로컬/세션 상태만. (예: React state + sessionStorage). 저장 API 없음.                                                                                                                                                                               |
| **노트**           | ❌ 미구현               | MSW 목업 또는 로컬 스토리지. 스튜디오별 1건 가정.                                                                                                                                                                                                  |
| **멤버 목록**      | ✅ 구현                 | `GET /api/studios/{studioId}/members` 연동. 응답 `ApiResponse<StudioMemberResponse[]>` 및 `role`: `owner` \| `admin` \| `member` 반영.                                                                                                             |
| **즐겨찾기(친구)** | ✅ 구현, 요청/수락 방식 | `GET /api/favorites`(래퍼 없음, `favorites[]`, `favoriteId`, `userId` 등), `GET /api/favorites/search?q=`, `POST /api/favorites`(userId로 요청), 요청 목록/수락/거절/삭제 API 연동. 기존 프론트 스키마와 필드명(`favoriteId`, `userId` 등) 맞추기. |
| **멤버 초대**      | ✅ 이메일 1명씩         | `POST /api/studios/{studioId}/members/invite` body `{ email, role: "MEMBER" \| "ADMIN" }`. 친구 목록에서 선택 시 해당 항목의 **email** 로 여러 번 호출.                                                                                            |
| **프라이빗 채팅**  | ✅ 채팅 API 재사용      | 같은 WebSocket 구독 + `platform === "INTERNAL"` 필터. 전송 시 `platform: "INTERNAL"`.                                                                                                                                                              |
| **녹화**           | ✅ 구현, title 없음     | Gateway: `POST /api/recordings/start`(body `studioId`), `POST /api/recordings/{studioId}/stop`, `GET /api/recordings/studio/{studioId}` 등. 제목 수정 UI는 비활성화 또는 추후 연동.                                                                |

## 3. 코드 수정·작성 예정 사항

### 3.1 공통

- **에러 응답**: `resultCode`, `message` 구조 반영해 API 클라이언트/에러 핸들링 통일.
- **Zod**: 백엔드에서 `null` 반환 가능 필드는 `.nullable().optional()` 사용.
- **즐겨찾기**: 응답이 래퍼 없음(`{ favorites, total, maxCount }` 등)인 엔드포인트는 기존 `apiClient.get` 후 `data` 가 아닌 `res.data` 또는 파싱 분기 처리.
- **Media 서비스**: 채팅·녹화는 동일 `NEXT_PUBLIC_API_URL`(Gateway 60000) 기준, 경로만 `/api/v1/media/...` 또는 `/api/recordings/...` 로 호출.

### 3.4 구현 순서 제안

1. **사이드바 공통**: `StudioSidebar` 탭 구성·활성 탭·패널 열기/닫기, 초기 비활성. `StudioMain`에서 `studioId` 전달.
2. **멤버**: 멤버 목록 API 연동, 멤버 패널, 초대 모달(이메일 + 친구 목록에서 email로 다중 초대).
3. **즐겨찾기 스키마/API**: 회신 스키마로 entities·apiClient 수정, 기존 즐겨찾기 화면과 스튜디오 초대 모달에서 동일 사용.
4. **녹화**: 녹화 API 연동, 녹화 패널(시작/중지/목록). 제목 수정 없음.
5. **채팅**: 히스토리 GET + 전송 POST 연동, 채팅 패널. WebSocket URL 확정 후 STOMP 구독 연동.
6. **프라이빗 채팅**: 채팅 패널 재사용, INTERNAL 필터 및 전송.
7. **배너·에셋·노트**: MSW 목업 + 패널 UI만 구현.
8. **스타일**: 로컬 상태 패널만 구현.

## 4. 정리

- **부족한 정보**: (1) 채팅 REST 경로(Gateway에서 `/api/v1/media/chat` vs `/api/media/chat`), (2) WebSocket(STOMP) 연결 URL, (3) 멤버 초대는 이메일만 사용하는지(친구 선택 시 email로 호출해도 되는지). 1·3은 문서만으로도 추정 가능해 우선 구현 후 조정 가능하고, 2는 실시간 채팅 연동 시 필수이므로 백엔드에 URL만 확인하면 됨.
- **코드 작업**: 위 “수정할 파일”·“새로 작성할 파일”·“구현 순서”대로 진행하면 되고, 회신에 명시된 스키마·에러 형식·nullable 처리·Gateway 경로를 그대로 반영하면 됩니다.
