# 프리뷰·송출 듀얼 스테이지 구조

> 스튜디오 미리보기 영역에서 **프리뷰용(컨테이너 크기)** 과 **송출/녹화용(고정 해상도)** Stage를 분리한 구조와 그 배경을 설명합니다.

---

## 1. 개요

스튜디오 페이지의 `PreviewArea`는 **두 개의 Konva Stage**를 사용합니다.

| 용도          | Stage        | 크기            | 표시 방식                              |
| ------------- | ------------ | --------------- | -------------------------------------- |
| **프리뷰**    | 표시용 Stage | 컨테이너에 맞춤 | 화면에 보임, 비율 유지(레터박스)       |
| **송출/녹화** | 캡처용 Stage | 720p/1080p 고정 | 화면 밖에 숨김, `captureStream()` 출력 |

둘 다 **같은 논리 좌표계**(720p: 1280×720, 1080p: 1920×1080)를 사용하고, **내용은 동일**합니다.  
차이는 **표시 방식**과 **출력 용도**뿐입니다.

---

## 2. 문제의 원인 (왜 듀얼 스테이지가 필요한가)

### 2-1. 요구사항 충돌

| 요구사항 | 프리뷰                                    | 송출/녹화                            |
| -------- | ----------------------------------------- | ------------------------------------ |
| **크기** | 브라우저 창·사이드바 등에 따라 **가변**   | **고정 해상도**(720p/1080p)          |
| **비율** | 컨테이너에 맞게 **스케일**, 레터박스 허용 | **1:1** 출력                         |
| **출력** | 화면에만 표시                             | `MediaStream`으로 전달 (라이브·녹화) |

프리뷰는 “보기 좋게” 컨테이너에 맞춰야 하고, 송출/녹화는 “일정한 해상도”로 내보내야 합니다.  
이 둘을 **하나의 Stage**로 처리하면 다음 문제가 생깁니다.

### 2-2. 단일 Stage로 처리할 때의 문제

1. **프리뷰에 맞추면**

   - Stage 크기 = 컨테이너 크기 (가변)
   - `captureStream()` → 해상도가 창 크기에 따라 달라짐
   - 송출/녹화 품질이 일정하지 않음

2. **송출에 맞추면**

   - Stage 크기 = 720p/1080p 고정
   - 창이 작을 때 프리뷰가 잘리거나, 크면 여백이 생김
   - UX가 나빠짐

3. **Stage 하나에 scale 적용**
   - 표시는 맞출 수 있으나, `captureStream()`은 **캔버스 실제 픽셀**을 그대로 캡처
   - scale을 바꿔도 캡처 해상도는 Stage의 width/height에 종속
   - 프리뷰와 송출의 목적이 달라 **역할 분리**가 필요함

## 3. 기존 구조 및 로직 (단일 Stage 가정)

### 3-1. 단일 Stage 시나리오

```
┌─────────────────────────────────────┐
│  Stage (단일)                        │
│  - width/height: 컨테이너 또는 고정?   │
│  - captureStream() → MediaStream     │
└─────────────────────────────────────┘
```

- Stage 하나가 **표시**와 **캡처**를 모두 담당
- 크기를 컨테이너에 맞추면 → 송출 해상도 불안정
- 크기를 고정하면 → 프리뷰 UX 저하

### 3-2. 논리 좌표계의 필요성

레이아웃·소스 위치·배너 등은 **고정된 논리 해상도**로 관리해야 합니다.

- `RESOLUTION_SIZE`: 720p(1280×720), 1080p(1920×1080)
- `sourceTransforms`, `arrangeSourcesInLayout` 등은 모두 이 좌표계 사용
- 창 크기가 바뀌어도 **내부 데이터는 변경하지 않고**, 표시할 때만 scale 적용

## 4. 변경 후 구조 및 로직 (듀얼 스테이지)

### 4-1. 전체 구조

```
PreviewAreaInner
├── 표시용 Stage (프리뷰)
│   ├── width/height: displayWidth, displayHeight (컨테이너 크기)
│   ├── pixelRatio: effectivePixelRatio (줌·DPR 반영)
│   └── Layer
│       └── Group (scaleX, scaleY, x, y)  ← 논리 좌표 → 컨테이너에 맞게 스케일
│           ├── 배경 Rect
│           ├── 소스들 (VideoSourceNode, ImageSourceNode 등)
│           ├── BannerOverlayNode
│           ├── AssetOverlayNode
│           └── Transformer (편집 모드)
│
└── 송출/녹화용 Stage (캡처)
    ├── width/height: stageWidth, stageHeight (720p/1080p 고정)
    ├── pixelRatio: 1 (출력 해상도 = 논리 해상도)
    ├── style: left: -9999, visibility: hidden
    └── Layer (captureLayerRef)
        └── 소스들, 배너, 에셋 (논리 좌표 그대로, scale 없음)
```

### 4-2. 표시용 Stage (프리뷰)

| 항목             | 값                                                                       | 설명                                                        |
| ---------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------- |
| **width/height** | `displayWidth`, `displayHeight`                                          | ResizeObserver로 측정한 컨테이너 크기                       |
| **pixelRatio**   | `effectivePixelRatio`                                                    | devicePixelRatio × visualViewport.scale (줌 시 선명도 유지) |
| **scale**        | `displayScale = min(displayWidth/stageWidth, displayHeight/stageHeight)` | 논리 해상도를 컨테이너에 맞게 축소                          |
| **offset**       | `displayOffsetX`, `displayOffsetY`                                       | 레터박스/필러박스 정렬                                      |

**로직 흐름:**

1. `containerRef`에 ResizeObserver 등록 → `containerSize` 갱신
2. `displayScale` = 컨테이너에 맞는 최대 scale (비율 유지)
3. `Group`에 `scaleX={displayScale}`, `scaleY={displayScale}` 적용
4. `displayOffsetX`, `displayOffsetY`로 중앙 정렬 (레터박스)

**결과:** 창 크기에 상관없이 논리 좌표계(720p/1080p)를 유지하면서, 화면에는 컨테이너에 맞게 표시됩니다.

### 4-3. 송출/녹화용 Stage (캡처)

| 항목              | 값                                  | 설명                  |
| ----------------- | ----------------------------------- | --------------------- |
| **width/height**  | `stageWidth`, `stageHeight`         | 720p 또는 1080p 고정  |
| **pixelRatio**    | `1`                                 | 논리 픽셀 = 출력 픽셀 |
| **위치**          | `left: -9999`, `visibility: hidden` | 화면 밖에 숨김        |
| **pointerEvents** | `none`                              | 이벤트 무시           |

**로직 흐름:**

1. 논리 좌표계 그대로 렌더 (scale 없음)
2. `captureLayerRef`의 canvas에 `captureStream(30)` 적용
3. `getPreviewStreamRef.current`로 이 MediaStream을 외부에 전달
4. 라이브 송출·녹화 시 이 스트림 사용

**결과:** 항상 720p 또는 1080p로 일정한 해상도의 MediaStream이 출력됩니다.

### 4-4. 공통 사항

- **논리 좌표계**: 두 Stage 모두 `stageWidth`, `stageHeight` 기준
- **소스 배치**: `arrangeSourcesInLayout`, `getTransform`으로 계산한 좌표를 그대로 사용
- **배너·에셋**: `BannerOverlayNode`, `AssetOverlayNode`에 `stageWidth`, `stageHeight` 전달
- **batchDraw**: 표시용 `scheduleBatchDraw`, 캡처용 `scheduleCaptureBatchDraw`로 각각 프레임당 1회만 그리기

## 5. 수정 과정 요약

### 5-1. 해결한 문제들

| 문제                    | 원인                              | 해결                                         |
| ----------------------- | --------------------------------- | -------------------------------------------- |
| 송출 해상도 불안정      | 단일 Stage가 컨테이너 크기에 종속 | 캡처 전용 Stage를 고정 해상도로 분리         |
| 프리뷰가 창에 맞지 않음 | 고정 해상도만 사용                | 표시용 Stage를 컨테이너 크기로, scale로 맞춤 |
| 역할 혼재               | 표시와 캡처가 한 Stage에 섞임     | 표시용 / 캡처용 Stage로 역할 분리            |

### 5-2. 적용한 설계 원칙

1. **논리 좌표계 고정**: 720p/1080p만 사용, 소스·배너·에셋은 모두 이 좌표계 기준
2. **표시와 출력 분리**: 표시는 scale로 조정, 출력은 1:1
3. **내용 동기화**: 두 Stage가 같은 `sortedSources`, `activeBanner`, `activeAsset` 등을 사용해 동일한 화면 구성

## 6. 결과 및 개선 사항

### 6-1. 달성된 결과

| 항목          | 내용                                                   |
| ------------- | ------------------------------------------------------ |
| **프리뷰**    | 컨테이너 크기에 맞게 표시, 비율 유지, 레터박스 적용    |
| **송출/녹화** | 720p 또는 1080p로 일정한 해상도 출력                   |
| **줌 대응**   | `effectivePixelRatio`로 브라우저 줌 시에도 선명도 유지 |
| **편집 UX**   | 논리 좌표 기반으로 드래그·리사이즈·스냅 동작 일관      |

### 6-2. 좋아진 점

1. **역할 분리**: 프리뷰와 송출이 서로 영향을 주지 않음
2. **해상도 일관성**: 송출/녹화 품질이 안정적
3. **반응형 프리뷰**: 창 크기 변경 시 자동으로 맞춤
4. **유지보수**: 표시 로직과 캡처 로직을 각각 수정 가능
